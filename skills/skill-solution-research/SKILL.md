# 研究方案规划

## 领域需求与流程调研

进入方案设计前，先调研研究项目的立项需求。学术与商业研究两界一致地把方案规划阶段的产出定义为"描述要做什么的文档"，而非研究结果本身（依据 Springer Bajare 2024 / PMI）。第一步必须通过需求澄清闸门，缺任一项不进入框架设计。

| 调研维度 | 要搞清的问题 | 调研方式 |
|----------|------------|----------|
| **研究目的分类** | 属于哪一类：探索性(What)/描述性(How many)/解释性(Why)/评估性(How well)？目的决定方法论方向。[依据: BCCampus / Creswell] | 与用户澄清 |
| **决策者与待支持决策** | 这次研究要支持哪个具体决策？谁基于结论拍板？（这是 Research Brief 的灵魂，所有权威来源强调"以决策为中心"）[依据: IFF / Beaton / ERIS] | 与用户确认 |
| **研究问题(FINER)** | 拆成 ≤5 个可回答的具体问题。用 FINER 自检：Feasible(可行)/Interesting(有价值)/Novel(有新意)/Ethical(合规)/Relevant(相关)。[依据: NIH PMC Covvey 2023 / Fandino 2019] | 与用户讨论 |
| **范围边界** | in-scope / out-of-scope 清单——明确包含什么、不包含什么，对治 scope creep。[依据: PMI / Research Brief] | 与用户确认 |
| **对标与已有研究** | 是否有同类研究/竞品分析可参考？避免重复劳动（FINER 的 Novel 维度）。[依据: FINER] | 派发 research_agent |
| **证据等级要求** | 关键结论需要多高可信度的证据？依据 OCEBM 证据金字塔：系统综述 > RCT > 队列 > 病例 > 专家意见。[依据: Oxford CEBM] | 与用户确认 |

**调研产出**：调研报告引用写入方案总纲；稳定框架写入 `.project/`（调研框架+维度+权重→`research_framework.md`，方法论选择+证据等级→`methodology.md`）。

## 方案内容框架

### 一、研究目标与范围

明确研究的核心导向：

| 维度 | 说明 | 示例 |
|------|------|------|
| **研究目标** | 核心研究问题或决策需求 | "评估3种分布式数据库在微服务架构下的适用性" |
| **研究类型** | 探索性/描述性/解释性/评估性 | 根据目标选择类型 |
| **范围边界** | 研究覆盖和排除的范围 | "只考虑开源方案，排除商业方案" |
| **预期产出** | 最终交付物形式 | 调研报告、对比矩阵、推荐方案 |
| **决策者** | 研究成果的受众和决策层级 | 技术负责人/产品经理/管理层 |

### 二、调研框架设计

定义调研的方法论和框架：

| 维度 | 说明 |
|------|------|
| **调研维度** | 从哪些维度进行对比分析（功能/性能/成本/生态/学习曲线等） |
| **评估标准** | 每个维度的评判标准和权重 |
| **数据来源** | 一手数据（实验/问卷/访谈）和二手数据（文档/社区/白皮书） |
| **样本范围** | 调研对象的选取范围和数量 |
| **时间窗口** | 调研的时间范围和数据时效性要求 |

**常见调研框架**：

1. **技术调研框架**：
   - 功能覆盖度 → 性能指标 → 生态成熟度 → 运维成本 → 学习曲线
   - 辅助：POC验证、基准测试

2. **市场调研框架**：
   - 市场规模 → 竞品格局 → 用户需求 → 商业模式 → 进入壁垒
   - 辅助：SWOT分析、波特五力

3. **学术研究框架**：
   - 文献综述 → 理论框架 → 研究方法 → 数据分析 → 结论建议
   - 辅助：假设验证、定量/定性分析

### 三、方法论选择

| 方法论 | 适用场景 | 优势 | 劣势 |
|--------|----------|------|------|
| **系统文献综述** | 学术研究、技术前沿追踪 | 全面、可复现 | 耗时、可能滞后 |
| **对比分析** | 技术选型、方案评估 | 直观、决策导向 | 维度选择影响结果 |
| **案例研究** | 深度理解特定对象 | 深入、context-rich | 难以泛化 |
| **实证研究** | 验证假设、测量效果 | 客观、可信度高 | 实验设计复杂 |
| **专家访谈** | 行业洞察、前沿判断 | 深度、前瞻性 | 主观性强 |
| **问卷调查** | 用户研究、需求收集 | 规模化、量化 | 设计偏差风险 |

### 四、分析维度与数据需求

根据研究目标定义具体的分析维度：

```
研究问题：{核心问题}

维度1: {维度名称}
├── 子维度1.1: {具体分析项}
├── 子维度1.2: {具体分析项}
├── 数据来源: {如何获取}
└── 分析方法: {如何分析}

维度2: {维度名称}
└── ...
```

**数据来源清单**（方案阶段列出，执行阶段采集）：

| 数据类型 | 来源 | 采集方式 | 可靠度 |
|----------|------|----------|--------|
| 官方文档 | 产品官网/技术文档 | 网页抓取/手动查阅 | ★★★★★ |
| 社区讨论 | GitHub/论坛/StackOverflow | 关键词搜索 | ★★★ |
| 基准测试 | 自有/第三方数据 | 实验运行/查阅报告 | ★★★★ |
| 行业报告 | Gartner/IDC/分析机构 | 付费/公开摘要 | ★★★★ |

### 五、产出物规划

研究类方案的核心产出物：

| 产出物 | 内容 | 受众 |
|--------|------|------|
| **调研报告** | 完整的研究过程和结果 | 决策者、执行者 |
| **对比矩阵** | 多方案的维度对比（表格形式） | 决策者 |
| **推荐方案** | 基于权重的综合推荐 | 决策者 |
| **数据附录** | 原始数据、详细分析过程 | 执行者 |

## 产出物拆分约定

研究类方案的产出**拆成三类**，各自职责不重叠。这是方案规划阶段必须遵守的产出模型。

### A. `.project/` 文档（稳定真相源，方案阶段定稿）

调研框架、方法论等稳定约定写入 `.project/`，**平铺不分子目录**，遵守 `document_context_rules.md`。
**禁止写入**调研结论原文、原始数据、逐条对比结果（那些是执行阶段产出，进 `docs/working/`）。

| 文件 | 装什么 |
|------|--------|
| `research_framework.md` | 研究目标与范围、调研维度与评估标准（含权重）、样本范围、时间窗口 |
| `methodology.md` | 方法论选择与理由、数据来源类型与可靠度等级、分析方法的统一约定 |

> 不是每个文件都必须有：方案涉及哪类信息就写哪类。无新内容的文件不创建。

### B. `docs/{title}_solution.md`（方案总纲，项目导航层）

总纲是项目导航与索引，服务于 `.project/` 文档和任务链。**只装导航层**：
- 研究背景与目标（探索/描述/解释/评估哪一类）
- 调研报告引用（指向 `docs/working/{title}_research_report.md` 路径，**不摘抄调研内容**）
- **讨论要点与关键研究决策**（讨论中确定的关键点、框架/方法论选择理由——为什么这么选，不是调研结论）
- **`.project/` 文件概览**（列出方案定稿了哪些 `.project/` 文件，每个一句话定位，让人知道去哪查调研框架/方法论等细节）
- 方案级 AC 总表（机读 YAML，侧重调研完整性/准确性/时效性）、变更记录、任务列表

调研框架细节、方法论、维度定义、调研内容摘抄**禁止放在这里**，它们下沉到 `.project/`、调研报告或任务文件。

### C. `docs/tasks/task_XX_*.md`（自包含任务文件）

任务文件必须自带「执行方案与设计」节，调研执行者拿到即可开工。研究领域该节写**调研契约级**内容：

| 必写项 | 说明 |
|--------|------|
| 本任务调研维度 | 本任务负责的分析维度与子维度（引用 `.project/research_framework.md`） |
| 数据来源 | 本任务的数据来源与采集方式、可靠度要求（引用 `.project/methodology.md`） |
| 分析方法 | 本维度如何分析、如何评分、如何归因 |
| 交付格式 | 本任务的产出格式与路径（对比矩阵/报告段落/数据附录），供后续汇总 |

**颗粒度**：到"调研什么维度、用什么方法、交付什么格式"，不到具体结论。让执行者知道"调什么、怎么调"，而非"结论是什么"。

## 注意事项

- 研究方案强调方法论的严谨性，比结果更重要的是获取结果的方式
- 每个分析维度必须有明确的评估标准，避免主观判断
- 数据来源必须标注可靠度等级（依据 OCEBM 证据金字塔：系统综述 > RCT > 队列 > 病例 > 专家意见）
- 方案阶段确定研究框架和维度，执行阶段按图索骥即可
- 研究类方案的任务链可能需要多个并行调研任务
- **方法必须与目的对齐**（Creswell 硬约束）：探索性目的→定性方法，因果目的→实验/纵向，评估目的→混合方法。方法-目的错配是最常见的反模式 [依据: Creswell / YouGov]
- **必须包含 Limitations 章节**：明确写"没覆盖什么、证据多强、哪些结论需谨慎"，学术与商业研究都强制要求 [依据: Creswell / Fandino]
- 对比/选型类调研必须公开维度权重（学 Forrester Wave），避免维度选择偏差 [依据: Forrester Wave 官方]
- 警惕**幸存者偏差和确认偏差**：竞品/案例调研强制纳入失败案例，预设"反驳证据"检索任务 [依据: The Decision Lab / Nickerson 1998]
