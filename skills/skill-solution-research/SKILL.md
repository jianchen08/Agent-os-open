# 研究方案规划

## 描述

研究类/调研分析类项目的方案规划技能。提供调研框架、研究方法论、分析维度、综合论证等研究领域特有要素的方案设计指导。替代统一模板中的通用章节，按学术研究和专业调研的维度组织方案内容。

## 适用场景

- 深度研究/复杂分析项目的方案规划阶段
- 调研框架设计、方法论选择的方案产出
- 市场调研/技术调研/学术研究/竞品分析等方向

## 方案内容框架

### 一、研究目标与范围

明确研究的核心导向：

| 维度 | 说明 | 示例 |
|------|------|------|
| **研究目标** | 核心研究问题或决策需求 | "评估3种分布式数据库在微服务架构下的适用性" |
| **研究类型** | 探索性/描述性/解释性/评估性 | 根据目标选择类型 |
| **范围边界** | 研究覆盖和排除的范围 | "只考虑开源方案，排除商业方案" |
| **预期产出** | 最终交付物形式 | 调研报告、对比矩阵、推荐方案 |
| **决策者** | 研究成果的受众和决策层级 | 技术负责人/产品经理/管理层 |

### 二、调研框架设计

定义调研的方法论和框架：

| 维度 | 说明 |
|------|------|
| **调研维度** | 从哪些维度进行对比分析（功能/性能/成本/生态/学习曲线等） |
| **评估标准** | 每个维度的评判标准和权重 |
| **数据来源** | 一手数据（实验/问卷/访谈）和二手数据（文档/社区/白皮书） |
| **样本范围** | 调研对象的选取范围和数量 |
| **时间窗口** | 调研的时间范围和数据时效性要求 |

**常见调研框架**：

1. **技术调研框架**：
   - 功能覆盖度 → 性能指标 → 生态成熟度 → 运维成本 → 学习曲线
   - 辅助：POC验证、基准测试

2. **市场调研框架**：
   - 市场规模 → 竞品格局 → 用户需求 → 商业模式 → 进入壁垒
   - 辅助：SWOT分析、波特五力

3. **学术研究框架**：
   - 文献综述 → 理论框架 → 研究方法 → 数据分析 → 结论建议
   - 辅助：假设验证、定量/定性分析

### 三、方法论选择

| 方法论 | 适用场景 | 优势 | 劣势 |
|--------|----------|------|------|
| **系统文献综述** | 学术研究、技术前沿追踪 | 全面、可复现 | 耗时、可能滞后 |
| **对比分析** | 技术选型、方案评估 | 直观、决策导向 | 维度选择影响结果 |
| **案例研究** | 深度理解特定对象 | 深入、context-rich | 难以泛化 |
| **实证研究** | 验证假设、测量效果 | 客观、可信度高 | 实验设计复杂 |
| **专家访谈** | 行业洞察、前沿判断 | 深度、前瞻性 | 主观性强 |
| **问卷调查** | 用户研究、需求收集 | 规模化、量化 | 设计偏差风险 |

### 四、分析维度与数据需求

根据研究目标定义具体的分析维度：

```
研究问题：{核心问题}

维度1: {维度名称}
├── 子维度1.1: {具体分析项}
├── 子维度1.2: {具体分析项}
├── 数据来源: {如何获取}
└── 分析方法: {如何分析}

维度2: {维度名称}
└── ...
```

**数据来源清单**（方案阶段列出，执行阶段采集）：

| 数据类型 | 来源 | 采集方式 | 可靠度 |
|----------|------|----------|--------|
| 官方文档 | 产品官网/技术文档 | 网页抓取/手动查阅 | ★★★★★ |
| 社区讨论 | GitHub/论坛/StackOverflow | 关键词搜索 | ★★★ |
| 基准测试 | 自有/第三方数据 | 实验运行/查阅报告 | ★★★★ |
| 行业报告 | Gartner/IDC/分析机构 | 付费/公开摘要 | ★★★★ |

### 五、产出物规划

研究类方案的核心产出物：

| 产出物 | 内容 | 受众 |
|--------|------|------|
| **调研报告** | 完整的研究过程和结果 | 决策者、执行者 |
| **对比矩阵** | 多方案的维度对比（表格形式） | 决策者 |
| **推荐方案** | 基于权重的综合推荐 | 决策者 |
| **数据附录** | 原始数据、详细分析过程 | 执行者 |

## 产出物拆分约定

研究类方案的产出**拆成三类**，各自职责不重叠。这是方案规划阶段必须遵守的产出模型。

### A. `.project/` 文档（稳定真相源，方案阶段定稿）

调研框架、方法论等稳定约定写入 `.project/`，**平铺不分子目录**，遵守 `document_context_rules.md`。
**禁止写入**调研结论原文、原始数据、逐条对比结果（那些是执行阶段产出，进 `docs/working/`）。

| 文件 | 装什么 |
|------|--------|
| `research_framework.md` | 研究目标与范围、调研维度与评估标准（含权重）、样本范围、时间窗口 |
| `methodology.md` | 方法论选择与理由、数据来源类型与可靠度等级、分析方法的统一约定 |

> 不是每个文件都必须有：方案涉及哪类信息就写哪类。无新内容的文件不创建。

### B. `docs/{title}_solution.md`（方案总纲，只讲 why）

不装调研结果。只装：研究背景与目标、调研摘要、**关键研究决策理由**（为什么用这个框架/方法论，不是调研结论）、方案级 AC 总表（机读 YAML，侧重调研完整性/准确性/时效性）、变更记录。
调研框架细节、方法论、维度定义**禁止放在这里**，它们下沉到 `.project/` 或任务文件。

### C. `docs/tasks/task_XX_*.md`（自包含任务文件）

任务文件必须自带「执行方案与设计」节，调研执行者拿到即可开工。研究领域该节写**调研契约级**内容：

| 必写项 | 说明 |
|--------|------|
| 本任务调研维度 | 本任务负责的分析维度与子维度（引用 `.project/research_framework.md`） |
| 数据来源 | 本任务的数据来源与采集方式、可靠度要求（引用 `.project/methodology.md`） |
| 分析方法 | 本维度如何分析、如何评分、如何归因 |
| 交付格式 | 本任务的产出格式与路径（对比矩阵/报告段落/数据附录），供后续汇总 |

**颗粒度**：到"调研什么维度、用什么方法、交付什么格式"，不到具体结论。让执行者知道"调什么、怎么调"，而非"结论是什么"。

## 使用场景

- 深度研究/调研项目的方案规划阶段
- 用户发起技术调研/市场研究/竞品分析需求时
- 方案规划助手识别到「任务类型=开放探索型」且领域为研究时

## 注意事项

- 研究方案强调方法论的严谨性，比结果更重要的是获取结果的方式
- 每个分析维度必须有明确的评估标准，避免主观判断
- 数据来源必须标注可靠度等级
- 方案阶段确定研究框架和维度，执行阶段按图索骥即可
- 研究类方案的任务链可能需要多个并行调研任务
