# 游戏方案规划

## 描述

游戏设计类项目的方案规划技能。提供游戏设计文档（Game Design Document）、数值体系、关卡结构、系统交互等游戏特有要素的方案设计指导。替代统一模板中的通用章节，按游戏设计的专业维度组织方案内容。

## 适用场景

- 游戏设计项目的方案规划阶段
- 游戏设计文档（GDD）的初步方案产出
- 数值策划/关卡策划/系统策划的方案设计

## 方案内容框架

### 一、游戏愿景与定位

明确游戏的核心定位：

| 维度 | 说明 |
|------|------|
| **游戏类型** | RPG/ACT/AVG/SLG/休闲/放置/卡牌等，以及混合类型 |
| **核心体验** | 一句话描述玩家核心体验（"让玩家感受到..."） |
| **目标用户** | 玩家画像（年龄/偏好/游戏习惯/付费习惯） |
| **平台** | 端游/手游/网页/主机/多平台 |
| **风格定位** | 美术风格、音乐风格、叙事风格 |
| **对标产品** | 参考竞品、差异化方向 |
| **商业模型** | 付费/免费/内购/订阅/广告 |

### 二、核心玩法设计

游戏的核心玩法和循环机制：

| 维度 | 说明 |
|------|------|
| **核心循环** | 玩家的基本行为循环（如：探索→战斗→收集→成长→探索） |
| **游戏机制** | 核心机制列表，每个机制的触发条件和结果 |
| **操作方式** | 输入方式（触屏/键鼠/手柄）、交互模式 |
| **反馈系统** | 视觉/听觉/数值反馈设计原则 |
| **难度曲线** | 从新手到精通的难度增长设计 |

**核心玩法设计原则**：
1. **易学难精**：入门门槛低，深度足够
2. **反馈即时**：每个操作都有明确反馈
3. **节奏感**：张弛有度，避免持续高强度或持续平淡
4. **策略深度**：玩家有做选择和优化的空间

### 三、数值体系

游戏的数值设计是游戏策划的核心：

| 维度 | 说明 | 设计要点 |
|------|------|----------|
| **属性系统** | 角色/物品/怪物的属性定义 | 属性间的克制关系、边际收益递减 |
| **成长曲线** | 玩家/角色的数值成长轨迹 | 平滑成长 vs 阶梯式突破 |
| **经济系统** | 货币体系、产出/消耗循环 | 防止通货膨胀、付费点设计 |
| **战斗公式** | 伤害计算/命中判定等关键公式 | 简单可预期、有策略变量 |
| **概率设计** | 抽卡/掉落/暴击等随机机制 | 保底机制、伪随机分布 |
| **平衡性** | PvP/PvE平衡、职业/流派平衡 | 动态平衡、版本调整策略 |

**数值设计原则**：
1. **可感知**：数值变化让玩家有明确的感受
2. **可预期**：玩家能大致预判数值变化带来的影响
3. **可优化**：给玩家留出数值优化的策略空间
4. **可控制**：运营可调节，有热更新/热修复的机制

### 四、系统设计

游戏各系统的功能定义和交互关系：

| 系统 | 核心功能 | 与其他系统交互 | 解锁条件 |
|------|----------|----------------|----------|
| **战斗系统** | 战斗流程、技能释放、AI行为 | 依赖属性系统、装备系统 | 初始可用 |
| **成长系统** | 升级、突破、技能树 | 消耗经济系统资源 | 初始可用 |
| **社交系统** | 好友、组队、公会、聊天 | 影响战斗（组队）、经济（交易） | 等级解锁 |
| **养成系统** | 装备/宠物/角色的收集和培养 | 消耗经济、影响数值 | 逐步解锁 |
| **活动系统** | 限时活动、日常/周常任务 | 产出资源、控制节奏 | 等级解锁 |

**系统交互图参考**：
```
经济系统 ←→ 养成系统 ←→ 战斗系统
   ↑                     ↓
   |←—→ 社交系统 ←—→ 成长系统
```

### 五、关卡与内容设计

| 维度 | 说明 |
|------|------|
| **关卡结构** | 章节/关卡/地图的层级关系 |
| **关卡目标** | 每个关卡的核心目标和玩法要素 |
| **场景设计** | 场景布局、障碍物、交互物、收集品 |
| **怪物配置** | 怪物类型、数量和AI行为模式 |
| **难度梯度** | 关卡间的难度递进策略 |
| **随机要素** | 地图随机生成/怪物随机配置/掉落随机 |
| **隐藏内容** | 彩蛋、隐藏关卡、收集要素 |

### 六、内容规划与生产任务

按内容类型划分生产任务：

| 内容类型 | 生产方式 | 预估量 | 优先级 |
|----------|----------|--------|--------|
| **美术资源** | 原画→模型→特效 | 按场景和角色估算 | 高 |
| **程序开发** | 功能模块实现 | 按系统和模块估算 | 高 |
| **数值配置** | 数值表和配表 | 按属性类型估算 | 中 |
| **文案剧情** | 剧情文本、角色对白 | 按章节估算 | 中 |
| **音效音乐** | 音效制作、配乐 | 按场景数量估算 | 低 |
| **测试验证** | QA测试、数值平衡测试 | — | 持续 |

## 产出物拆分约定

游戏类方案的产出**拆成三类**，各自职责不重叠。这是方案规划阶段必须遵守的产出模型。

### A. `.project/` 文档（稳定真相源，方案阶段定稿）

核心玩法、数值公式、系统交互等稳定设计写入 `.project/`，**平铺不分子目录**，遵守 `document_context_rules.md`。
**禁止写入**具体关卡配置表、美术资源清单、逐场景配置。

| 文件 | 装什么 |
|------|--------|
| `game_design.md` | 游戏愿景与定位、核心循环、核心玩法机制、操作方式、反馈系统、难度曲线 |
| `numerics.md` | 属性系统定义、成长曲线与公式、经济系统（货币/产出消耗）、战斗公式、概率与保底机制 |
| `systems.md` | 各系统功能定义、系统间交互关系与数据流向、解锁条件、系统级约束 |

> 不是每个文件都必须有：方案涉及哪类信息就写哪类。无新内容的文件不创建。

### B. `docs/{title}_solution.md`（方案总纲，只讲 why）

不装具体数值和系统细节。只装：游戏愿景与定位（类型/核心体验/目标玩家/对标）、调研摘要、**关键设计决策理由**（为什么这么设计，不是设计本身）、方案级 AC 总表（机读 YAML）、变更记录。
核心玩法机制、数值公式、系统设计**禁止放在这里**，它们下沉到 `.project/` 或任务文件。

### C. `docs/tasks/task_XX_*.md`（自包含任务文件）

任务文件必须自带「执行方案与设计」节，执行者拿到即可开工。游戏领域该节写**设计契约级**内容：

| 必写项 | 说明 |
|--------|------|
| 本任务数值公式 | 本系统/模块涉及的公式、数值范围（引用 `.project/numerics.md`） |
| 触发条件与结果 | 本机制何时触发、触发后系统状态如何变化 |
| 系统接口 | 与其他系统的交互接口（产出/消耗什么、监听什么事件，引用 `.project/systems.md`） |
| 核心约束 | 不可违反的设计原则（如平衡性、确定性、可热更） |

**颗粒度**：到"公式和触发契约"，不到具体配表数值和实现。让执行者知道"实现什么机制、满足什么数值约束"，而非"填哪些表"。

## 使用场景

- 游戏项目启动时，方案规划阶段使用
- 用户发起游戏设计/游戏开发需求时
- 方案规划助手识别到「任务类型=精确创意型/开放创意型」且领域为游戏时

## 注意事项

- 数值设计必须给出明确的公式和数值范围，不能模糊
- 核心玩法必须在方案阶段定稿，后续只做增量不重构
- 美术风格和概念设计建议先做打样验证
- 关卡设计在方案阶段给出结构即可，具体配置可在执行阶段细化
