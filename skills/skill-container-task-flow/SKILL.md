# 容器任务流程

## 适用场景

L1 创建和管理容器任务（一个 app、一个游戏、一部网络小说等复杂长期任务）时加载本 skill。

## 一、容器创建流程

1. **澄清需求**
2. **创建容器**：使用 `task_submit(task_scope="container")` 创建容器任务
   - **容器不能指定 target_type 和 target_id**
   - 创建后获得容器 ID，后续步骤的所有任务提交都必须传 `parent_task_id=容器ID`
3. **设置进度触发器**：使用 `trigger_setup` 设置定时触发器（详见下文）
4. **提交规划任务**：派发给 `solution_planning_agent`
5. **执行任务链**（方案确认后立即执行，无需询问用户）：
   - **直接传递方案路径**：不要自己读任务链再转述给下级，下级比你更专业，它自己会读。goal 中写明方案文档路径（如 `docs/{title}_solution.md`），让下级自行读取任务链并执行
   - 用 `file_read(fields=["task_chain"])` 只读取任务链中的**执行者列表**，用于检查所需 Agent 是否已存在
   - 缺失资源 → 触发资源准备流程创建
   - 等待资源就绪后，将方案路径提交给对应执行者
6. **容器完成**：任务链全部完成（含最后的端到端验证任务）后，先通知用户容器已完成，再用任务管理工具标记容器完成
   - **取消触发器**：容器完成后立即使用 `trigger_setup(action="cancel", trigger_id="之前记录的ID")` 取消进度触发器
   - 注意：用任务管理工具标记完成是完成容器的唯一方式

## 二、workspace 决策

| 场景 | workspace 填法 |
|------|---------------|
| 全新项目（一般情况） | 不填 |
| 用户明确指定修改某个已有项目目录 | 填该目标项目目录 |
| ⚠️ 系统内部路径（container_xxx） | 禁止填写，只用于资源管理任务 |

## 三、进度触发器配置

使用 `trigger_setup` 设置定时触发器，定期检查容器执行进度。

| 配置项 | 值 |
|--------|-----|
| **触发间隔** | 15 分钟 |
| **最大触发次数** | 小型 32 次（约 8 小时）、中型 96 次（约 24 小时）、大型 288 次（约 3 天） |
| **触发动作** | 检查容器子任务当前状态，对照速查表验证产出物是否已生成 |
| **共用触发器** | 已有相同间隔的活跃触发器时不要新建，用 update 延长其生命周期 |
| **超时处理** | 触发次数耗尽后容器仍未完成 → 延长次数并通知用户 |
| **记录 trigger_id** | 容器完成时用于取消 |

**停滞检测**：子任务运行超 1 小时时查看详细进度，下次触发对比是否有变化。有变化继续等，无变化按优先级处理：①注入提示引导恢复 ②分析错误尝试修复 ③多次无效再通知用户。

## 四、方案不可变原则

方案准备阶段人类确认的方案稿是任务执行的**绝对标准**，确认后不可更改：

- **方案内容不可变**：已确认方案的核心设计、功能范围、技术选型等不可修改
- **细化发现问题**：细化过程中发现方案有缺陷或需要调整，细化 Agent 需提交新的细化任务说明问题，由 L1 决定处理方式
- **执行中发现方案需要更改**：必须向人类请示，获得明确同意后方可修改方案
- **细化细节可调整**：方案细化的粒度、实现步骤等细节可以在不违背方案核心的前提下调整

### 架构调整与方案修订

容器任务执行过程中，可能出现需要调整方案核心架构的情况。此时必须经过严格审查：

1. **判断影响范围**：架构调整是否涉及已确认方案的核心设计、功能范围或技术选型
2. **如果涉及核心变更**：
   - 必须先向人类请示，获得明确同意
   - 或提交给下级 Agent（如 solution_planning_agent）进行审查评估，产出审查报告
   - 人类根据审查报告决定是否同意修订
3. **执行修订**：获得同意后，重新提交 `solution_planning_agent`（继承原容器工作空间），让其基于已有方案进行修订，产出更新后的方案文档和任务链
4. **人类确认修订**：修订后的方案同样需要人类确认
5. **继续执行**：根据新任务链继续派发，未受影响的已完成子任务无需重做

**判断标准**：
- 核心变更：技术选型改变、核心功能增删、架构模式变更
- 非核心变更：实现细节调整、性能优化、代码重构（不改变接口）

## 五、失败处理与工作空间规则

### 工作空间规则

- **资源管理任务**（提交给 resource_manager_agent）需要设置 `workspace="<项目根目录绝对路径>"`
- workspace 参数含义、子任务继承、inherit_workspace_from 等通用规则参见 task_submit 工具描述

### 失败处理原则

完整失败处理策略见 task_manage 工具描述。核心原则（必须遵守）：

- **优先抢救已有成果，禁止动辄重新提交**：任务失败（尤其合并失败）时，第一反应是"看失败原因 + 检查 worktree 有没有产出"，不是"重新提交"。子任务在 worktree 中完成的工作是真实产出，丢弃重做是最大浪费
- **合并失败用 resource_merge 合并**：子任务代码写好了但 worktree 合并冲突，用 resource_merge 工具（action=merge）将 worktree 变更合并到目标空间，禁止重新提交丢弃成果
- **有产出就抢救，无产出才重做**：worktree 有有效产出 → non_isolated 模式/continue 定向修复；无产出或完全错误 → 才重新提交（须继承原工作空间）

### 隔离模式区分

评估失败时的处理方式取决于子任务的隔离模式：

| 模式 | 失败特征 | 处理方式 |
|------|---------|---------|
| **non_isolated** | 改动已直接生效到原文件，评估失败只是验收指标问题 | 确认产出无误即可，不需要重新提交 |
| **isolated** | worktree 不会自动合并到目标空间 | ①手动用 resource_merge 合并 worktree 变更到目标空间，或②重新提交让评估通过后自动合并 |

### 角色特定原则

- **Agent 不可更换**：任务链中指定的 Agent 不可更换，任何情况下都不考虑换 Agent
- **禁止跳步**：重新提交必须保持原流程位置不变
- **兜底**：多次处理仍失败后必须通知人类，不要自行放弃或绕过
- 人类确认无法解决 → 用任务管理工具标记容器失败

## 六、容器管理规则

- 只有所有子任务都成功完成且文件产出符合验收标准，且已通知用户后，才用任务管理工具标记容器完成
- 收到通知后立即处理，不要攒着等全部完成
- 无依赖的子任务可以并行提交，有依赖的必须等前置完成
- **容器完成需用户确认**：标记 completed 前必须获得用户明确确认，不能自行判断
- **容器内禁止裸提交**：容器生命周期内所有操作都必须作为容器子任务提交
- **容器=项目目录管辖权**：容器绑定项目目录后，对该目录的任何读写操作都必须走容器子任务
- **关联工作时必须继承**：新任务与已有任务相关联时必须继承（优先 pipe+workspace，只能继承一个时至少 workspace，容器子任务间继承 pipe）；不相关则不需要
