# 软件方案规划

## 领域需求与流程调研

进入方案设计前，先调研软件项目的需求与约束，确保方案只锁定"有真实权衡的关键决策"，把无权衡的细节推迟到执行阶段（依据 Google Design Doc 实践）。

| 调研维度 | 要搞清的问题 | 调研方式 |
|----------|------------|----------|
| **需求来源与背景** | 是新项目、功能新增、重构还是 Bug 修复？需求方是产品/技术/用户反馈？需求来源是 PRD、User Story、技术债还是合规要求？[依据: Atlassian PRD / Google Design Doc] | 派发 research_agent + 与用户澄清 |
| **现有上下文** | 是否有现成代码/架构？技术栈是什么？哪些必须复用、哪些可替换？遗留约束（历史决策、外部依赖）有哪些？系统上下文边界在哪？[依据: arc42 §3 Context / Google Design Doc §1] | file_read 现有 `.project/` 文档与代码结构 |
| **目标与非目标** | Goals（做什么）和 Non-goals（本可以做但明确不做的事）是什么？非目标不是"不要崩溃"，而是"本可以做但选择不做"，用于防止范围蔓延。[依据: Google Design Doc] | 与用户讨论确认 |
| **非功能需求(NFR)** | 性能、可用性、安全、可扩展性等质量属性——每个量化到可验证场景（刺激源→刺激→响应→响应度量）。NFR 是架构约束，直接限制设计决策。[依据: ATAM / Red Hat NFR / arc42 §2] | 派发 research_agent + 用户确认 |
| **技术约束** | 必须用/不能用的技术栈？必须集成的遗留系统/外部依赖？部署环境约束？[依据: arc42 §2 Constraints] | 与用户确认 |
| **关键决策点** | 哪些决策"有真实权衡可言"（值得写 design doc / ADR）？哪些明显无权衡（该直接写代码）？[依据: Google Design Doc / ADR(Nygard)] | 与用户讨论 |

**调研产出**：调研报告引用写入方案总纲；关键约束和架构决策写入 `.project/`（架构约束→`architecture.md`，功能边界→`features.md`，决策记录→ ADR 形式附于 `architecture.md`）。

## 方案内容框架

### 一、架构设计

系统整体架构必须明确以下维度：

| 维度 | 说明 | 决策方式 |
|------|------|----------|
| **分层架构** | 系统层次划分（展示层/业务层/数据层/基础设施层等） | 根据复杂度选择分层数 |
| **模块划分** | 按业务领域划分子模块，明确各模块职责边界 | 领域驱动/功能聚合 |
| **数据流** | 模块间的数据流动方向、数据格式、传输方式 | 根据耦合要求选择 |
| **部署架构** | 服务部署方式、环境划分、基础设施依赖 | 根据规模和可用性要求选择 |

**架构设计原则**：
1. **关注点分离**：各模块职责单一，高内聚低耦合
2. **可演化性**：架构支持增量演进，避免过度设计
3. **约定优于配置**：标准化接口和通信方式
4. **防御性设计**：考虑异常情况、错误边界和降级策略

### 二、技术选型

技术选型的方案中应包含对比分析：

| 决策点 | 备选方案 | 评估维度 | 推荐方案 |
|--------|----------|----------|----------|
| **语言/运行时** | Python/Node.js/Go/Java/Rust等 | 生态成熟度、性能要求、约束兼容性 | — |
| **Web框架** | FastAPI/Express/Spring/Flask等 | 开发效率、性能、生态 | — |
| **数据库** | PostgreSQL/MySQL/MongoDB/Redis等 | 数据模型、一致性要求、查询模式 | — |
| **前端框架** | React/Vue/Svelte/Next.js等 | 项目规模、SSR需求、生态成熟度 | — |
| **部署方式** | Docker/K8s/Serverless/VPS | 运维复杂度、成本、扩展性要求 | — |
| **测试框架** | Pytest/Vitest/Jest/Playwright等 | 语言匹配、社区活跃度 | — |

**技术选型原则**：
1. **成熟优先**：优先选择经过验证的成熟技术栈
2. **约束适配**：符合项目的技术约束和遗留系统兼容性
3. **生态完整**：文档、社区、第三方库支持要好
4. **避免锁定**：关键决策点避免绑定特定供应商

### 三、模块与接口设计

将系统拆分为可独立执行的模块，定义模块间接口：

| 模块 | 职责 | 内部结构 | 对外接口 | 依赖 |
|------|------|----------|----------|------|
| 模块A | 核心职责描述 | 子组件划分 | REST/GraphQL/Event API | 依赖B |
| 模块B | 核心职责描述 | 子组件划分 | 提供接口说明 | 无 |

**接口设计规范**：
1. **RESTful 规范**：资源命名、HTTP方法、状态码统一
2. **输入验证**：Schema定义、参数校验规则
3. **错误处理**：统一错误格式、错误码定义
4. **版本控制**：API版本策略（URL路径/Header/参数）

### 四、数据模型

数据结构设计，方案中定义核心实体：

```
实体：User
├── id: UUID (PK)
├── username: string (unique)
├── email: string
├── password_hash: string
├── role: enum [admin, user, viewer]
├── created_at: datetime
└── updated_at: datetime

实体：Project
├── id: UUID (PK)
├── name: string
├── owner_id: UUID → FK(User.id)
└── status: enum [active, archived]
```

**数据模型原则**：
1. **范式化**：减少数据冗余，保持一致性
2. **可索引**：核心查询字段设计合理索引
3. **可扩展**：预留扩展字段或机制
4. **审计记录**：关键变更记录时间戳和操作人

### 五、测试策略

方案中测试策略的定义：

| 测试层级 | 覆盖范围 | 工具/框架 | 执行频率 |
|----------|----------|-----------|----------|
| **单元测试** | 核心逻辑、业务规则 | Pytest/Jest | 每次提交 |
| **集成测试** | API接口、数据库操作 | Pytest-asyncio/Supertest | 每日 |
| **E2E测试** | 用户旅程、关键流程 | Playwright/Cypress | 每次发布 |

**测试蓝图**（必须包含，写入方案文档的测试蓝图章节）：
- **用户旅程表格**：角色→操作→前置条件→期望结果
- **影响矩阵表格**：操作→前端影响→后端影响→副作用

## 产出物拆分约定

软件类方案的产出**拆成三类**，各自职责不重叠。这是方案规划阶段必须遵守的产出模型。

### A. `.project/` 文档（稳定真相源，方案阶段定稿）

架构决策、接口契约、领域模型等稳定信息写入 `.project/`，**平铺不分子目录**，遵守 `document_context_rules.md`。
**禁止写入**代码示例、文件目录树、模块清单、小功能描述。

| 文件 | 装什么 |
|------|--------|
| `architecture.md` | 分层架构、模块边界与职责、部署架构、跨模块数据流方向 |
| `api_contract.md` | 模块/服务间的接口契约：方法签名、请求/响应结构、错误码、版本策略 |
| `domain_model.md` | 核心实体的字段定义、实体间关系、关键不变式与业务规则 |
| `features.md` | 功能清单与边界（做什么、不做什么）、功能间依赖，不含实现细节 |

> 不是每个文件都必须有：方案涉及哪类信息就写哪类。涉及新架构决策写 `architecture.md`，涉及新接口契约写 `api_contract.md`，以此类推。无新内容的文件不创建。

### B. `docs/{title}_solution.md`（方案总纲，项目导航层）

总纲是项目导航与索引，服务于 `.project/` 文档和任务链。**只装导航层**：
- 背景与目标（为什么做）
- 调研报告引用（指向 `docs/working/{title}_research_report.md` 路径，**不摘抄调研内容**）
- **讨论要点与关键决策**（讨论中确定的关键点、技术选型/架构决策的理由——为什么选这个，不是方案本身）
- **`.project/` 文件概览**（列出方案定稿了哪些 `.project/` 文件，每个一句话定位，让人知道去哪查架构/接口/数据模型等细节）
- 方案级 AC 总表（机读 YAML）、测试蓝图（用户旅程 + 影响矩阵，编程类必填）、变更记录、任务列表

架构设计、模块接口、数据结构、实施步骤、调研内容摘抄**禁止放在这里**，它们下沉到 `.project/`、调研报告或任务文件。

### C. `docs/tasks/task_XX_*.md`（自包含任务文件）

任务文件必须自带「执行方案与设计」节，执行者拿到即可开工。软件领域该节写**接口契约级**内容：

| 必写项 | 说明 |
|--------|------|
| 对外接口签名 | 本任务产出的对外 API（方法名、参数、返回、错误情况） |
| 依赖接口 | 调用的其他模块/服务接口（引用 `.project/api_contract.md` 或给出签名） |
| 关键数据结构 | 本任务涉及的核心实体字段（引用 `.project/domain_model.md`） |
| 核心决策与约束 | 本任务的技术选型、不可违反的约束 |

**颗粒度**：到接口签名和字段定义，不到步骤、伪代码、具体实现。让执行者知道"做什么、对外的契约是什么"，而非"怎么做"。

完整任务文件结构见方案规划助手的「任务链」章节。

## 注意事项

- 技术方案必须有可验证的验收标准（支持二值判定）
- 架构决策必须说明选择理由和替代方案对比，以 ADR 形式记录（Context / Decision / Consequences）[依据: Nygard / Fowler]
- 测试蓝图是编程类项目的必需产出
- **Monolith First**：先用单体理解领域边界，再考虑拆微服务；服务边界应对齐 DDD 限界上下文，不按技术任意切分 [依据: Fowler & Lewis]
- **YAGNI**：只为当前需求实现，不为"可能的未来需求"实现；只锁定有真实权衡的决策，无权衡的细节推迟到执行阶段 [依据: Google Design Doc]
- NFR 是架构约束而非事后补丁，必须在方案阶段量化为可验证场景 [依据: ATAM]
