# 代码调试

## 总原则

**先确认"谁在执行"，再优化"怎么执行"。** 报错位置常常是受害者，不是凶手。

### 强制流程（跳步视为流程违规）

```
写失败测试（规则 0）
  → 收集数据（规则 8）
  → 分析根因（规则 7 / 9 / 12）
  → 写回归测试并验证（规则 11）
  → 修改代码
```

本技能遵循《编程领域规则》中的行为测试规范（§6.3 行为测试原则、§6.4 时序测试铁律）。

---

## 执行流程

### 第 1 步：确认需要（调试前必做）

进入调试前，先确认故障信息和复现条件就位，缺则停下追问，不靠猜测推进。

1. **确认故障现象**：报的什么错、什么场景下出现、预期行为 vs 实际行为
2. **确认复现条件**：最小复现路径、触发输入、运行环境；无法复现的 bug 先和上报方确认复现步骤
3. **确认需求维度**：这个 bug 是"功能没实现/行为不符合预期"还是"报错崩溃"。前者是**需求符合性问题**——核对实际行为偏离了哪个 AC、设计本意应该是什么，不能只盯着报错；后者才是纯技术故障
4. **确认涉及范围**：故障在哪个模块、调用链上下游、可能受影响的依赖
5. **确认已有线索**：错误日志、stack trace、已知的可疑点、之前是否修过该模块
6. **信息不全则停下**：通过 human_interaction 向上追问，确认齐了再进入第 2 步

### 第 2 步：摸清现有实现（防重复修复、防破坏架构，调试前必做）

调研故障模块现有实现，需要得出以下结论：

- 调用链摘要：故障模块的调用链上下游、数据流、依赖方向
- 可复用项：现有代码里有没有已经正确处理该场景的函数/分支
- 接口契约：本模块对外接口的签名、入参出参、异常、调用约束
- 架构边界：本模块职责边界、依赖方向，修复会不会跨过模块边界乱改、破坏对外接口契约

### 第 3 步：复现与定位（最多 2 轮，只加诊断标记，禁止改代码）

按 coding_domain_rules.md §1.5 Bug修复安全执行。

1. **先写失败测试**（见规则 0）：必须先写一个能复现该 bug 的行为测试，确认它失败。已有测试能复现则跳过这步。
2. **最小输入复现**：用最小的输入触发失败
3. **收集数据**（见规则 8）：按数据收集清单采集事实，不靠直觉
4. **确认 WHO 不只是 WHAT**（见规则 1）：确认"是哪个对象/实例在执行"，而不只是"报了什么错"

### 第 4 步：追踪根因（最多 2 轮，只加诊断标记，禁止改代码）

1. **逐层追问根因**（见规则 7）：用 5 Whys 追到可操作根因
2. **管理假设**（见规则 9）：同时跟踪不超过 3 个假设，各列正反证据
3. 在构造函数加 `traceback`（limit=4）
4. 确认对象被创建了几次、由谁创建
5. 检查 fork/clone/rebuild 是否丢失依赖
6. **检查对象生命周期**：bug 通常在对象被重建时丢失属性，不在报错代码本身

### 第 5 步：修复根因

1. 修复**真正的根因**（不是症状）
2. 在**完整系统中验证**（不是只跑隔离测试——见规则 2）
3. 移除所有临时诊断代码（print/traceback 标记）

### 第 6 步：回归验证

1. **写回归测试**（见规则 11）：必须是行为测试，验证根因被修复而非症状消失
2. 重新运行**全部测试**（包括之前通过的）
3. 复验失败测试：原来复现 bug 的测试现在应通过
4. 如果仍有失败 → 回到第 3 步（同方向最多 3 次，超过则切换假设）

## 调试原则（含使用条件与示例）

### 规则 0：用测试捕获理想行为（前置硬门槛）

**使用条件**：发现 bug 的第一时间——任何分析代码、读日志、改逻辑之前。

**怎么使用**：
- 第一动作是写出能暴露 bug 的行为测试（红灯），而不是分析代码。该测试即**理想模型的可执行形式**——精确描述"在给定输入下，正确的可观察行为应该是什么"。
- 优先于所有分析动作。先有失败测试，再谈诊断。
- 测试必须断言**可观察行为**（输入 → 输出 / 副作用），而非内部实现细节，遵循《编程领域规则》§6.3。
- 如果无法写出测试（缺乏可观测性），**不要绕过这一步**——先重构增加可观测性（暴露事件、时间戳、状态查询接口），再写测试。

**小例子**：
> 通知推送时序错了，先不读代码。直接写一个测试断言"通知推送时间戳 ≤ LLM 收到请求时间戳"，跑一次，红灯——这就是 bug 的可执行形式。

### 规则 1：自顶向下，定位执行实例

**使用条件**：拿到报错的第一时刻——尤其是有明确的异常类型/报错信息时。

**怎么使用**：先定位**执行实例**（是哪个对象在跑），再定位代码行。报错位置常常是受害者，不是凶手。在所有可能的分支/实现都加标记（含 `type(self).__name__`），而不是只在你"认为"正确的那个加。

**小例子**：
> 家里灯不亮，直接拆灯泡检查——但可能根本是电闸跳了。先确认"是这一盏灯不亮，还是整层楼不亮"（定位范围），再决定查灯泡还是查电闸。

### 规则 2：隔离测试通过 ≠ 系统正确

**使用条件**：隔离测试（单元测试）通过了，但完整系统仍然失败时。

**怎么使用**：隔离层验证"逻辑正确性"，系统层验证"编排正确性"，二者不等价。隔离测试通过绝不等于 bug 已修复。通过后如果 bug 仍在，**立刻停止在隔离层继续修改**，转向系统集成层。经验触发线：最多再做 1 轮隔离测试确认，然后必须切全链路追踪。

**小例子**：
> 汽车零件单独检测都合格，但组装后仍可能发动不了——零件正确不等于装配正确。

### 规则 3：追踪对象生命周期

**使用条件**：功能 A 在隔离测试中正常，但在系统中失败——典型的"代码对、运行时错"。

**怎么使用**：追踪**实例的来源与重建路径**——它被构造了几次、由谁构造、运行时用的实例和你修改的是否是同一个。检查所有工厂 / fork / clone / rebuild 路径。

**小例子**：
> 你以为在改张三的档案，但系统里其实有两个"张三"。改了半天没用，因为运行时用的是另一个张三。先确认手里这个张三的 id，再看它从哪个入口被创建。

### 规则 4：诊断加在分叉点，不是执行点

**使用条件**：需要加日志诊断时——尤其在有多态、策略模式、插件注册表的系统里。

**怎么使用**：日志加在**分叉点**（多个实现可能被执行的地方），而不是执行点（具体实现内部）。分叉点选错了实现，执行点内的日志全是在错误代码上打的，毫无价值。

**小例子**：
> 十字路口有四条岔路，不知道车开去了哪条。正确做法是在路口（分叉点）装一个摄像头，而不是在某一条岔路上装 20 个。

### 规则 5：主动证伪，警惕 confirmation bias

**使用条件**：每次修改后 bug 仍在时——防止陷入"越改越觉得快好了"的死循环。

**怎么使用**：优先找**推翻**假设的证据，而不是找支持它的证据。每当修改后 bug 仍在，问自己："如果我的假设是错的，最可能的替代解释是什么？" 经验触发线：同一方向尝试 3 次无效后，**强制切换假设**——在偏见固化前打断。

**小例子**：
> 医生只找支持"感冒"诊断的症状，会漏诊肺炎；好医生会专门找能推翻"感冒"的证据——发烧到几度？持续几天了？有没有胸痛？

### 规则 7：根因分析 — 逐层追问（5 Whys）

**使用条件**：每次修复后 bug 仍在时；或发现自己开始在同一方向打转时。

**怎么使用**：
- 从表层症状逐层追问"为什么"，追到**可操作根因**为止——即改了它，bug 不会以任何形式复现。
- 典型因果链深度 4-5 层，每层各追一类信息：

| 追问层 | 从 ___ 追到 ___ | 回答的问题 |
|--------|----------------|-----------|
| Y1 | 症状 → 直接原因 | 表面错误是什么导致的？ |
| Y2 | 直接原因 → 触发条件 | 这个直接原因在什么条件下发生？ |
| Y3 | 触发条件 → 运行时机制 | 系统运行时哪个机制制造了这个条件？ |
| Y4 | 机制 → 设计选择 | 这个机制为什么这样设计 / 漏了什么？ |
| Y5 | 设计 → 根本原因 | 根本的设计 / 流程缺陷是什么？ |

- **终止条件**（比"问够 5 次"更重要）：到达可操作根因即停，不必凑满 5 次；3 层追到根因就停。超过 5 层仍无根因 → 转规则 12（鱼骨图，多维度并行排查）。
- 每一层的答案必须是**事实**（日志/堆栈/数据），不是猜测——偏差会跨层累积。

**小例子**：
> 迟到 →（Y1）起晚了 →（Y2）闹钟没响 →（Y3）手机昨晚被设成静音 →（Y4）开会时手动静音忘了恢复。到 Y4 就是**可操作根因**：建立"会议结束自动恢复铃声"的流程，bug 根除。不必硬凑 Y5。

### 规则 8：先收集数据，再分析

**使用条件**：形成任何假设之前——尤其是想"我觉得问题出在……"的时候。

**怎么使用**：假设必须基于**事实数据**，而不是直觉。数据收集阶段不改代码（只加诊断）。经验触发线：至少收集 2 轮数据后再形成假设。

**数据收集清单**：

| 数据类型 | 收集方式 | 目的 |
|---------|---------|------|
| 完整错误堆栈 | 捕获 traceback | 定位出错位置 |
| 对象类型和ID | `print(type(x).__name__, id(x))` | 确认是哪个实例 |
| 输入参数快照 | 打印关键参数值 | 排除参数错误 |
| 执行路径标记 | 在分叉点加 print | 确认走了哪条路径 |
| 对象创建栈 | `traceback.print_stack(limit=4)` | 追踪对象生命周期 |

**小例子**：
> 医生先做检查（验血、拍片）再下诊断，不会瞄一眼就开药。

### 规则 9：证据驱动的假设管理

**使用条件**：开始形成多个假设，需要决定优先验证哪个时。

**怎么使用**：
- 假设收敛、证据驱动。每个假设同时跟踪正面和反面证据；**排除假设比确认假设更有价值**——被排除的假设不用再试。
- 经验触发线：同时维护的假设**不超过 3 个**。如果 3 个假设都被排除，不要硬造第 4 个，而是**回到数据收集阶段**（规则 8）——大概率遗漏了关键信息。

**假设模板**：

```
假设 H1：<描述>
  正面证据：<支持此假设的事实>
  反面证据：<与此假设矛盾的事实>
  验证方式：<如何证明/证伪>
  状态：待验证 / 已排除 / 已确认
```

**小例子**：
> 侦探不会同时追 10 个嫌疑人，而是聚焦 2-3 个最可疑的，分别列出"支持是他"和"推翻是他"的证据。排除一个嫌疑人比锁定一个更有价值。

### 规则 10：架构级熔断器

**使用条件**：修复尝试反复失败、开始怀疑问题不在代码层时。

**怎么使用**：
- 当修复尝试反复失败时，问题大概率不在代码层而在架构层。失败累积到阈值，**停止改代码，升级到架构重新评估**。
- 经验熔断线（满足任一即触发）：同一代码位置修改约 3 次仍未通过 / 同一假设方向尝试约 3 次无效 / 总修复轮次累计约 6 次仍未解决。
- 熔断后：停止所有代码修改 → 重新评估架构 → 标记"bug 发生位置"和"修复尝试位置"→ 若两者不在同一层，问题在架构层而非代码层。画图是辅助手段，非强制产物。

**小例子**：
> 修水龙头连换 3 个垫圈还漏水，该停下想"是不是水管接口裂了 / 水压太高"，而不是继续买第 4 个垫圈。

### 规则 11：回归测试的正确写法

**使用条件**：定位到根因、准备写回归测试时。

**怎么使用**：
- 回归测试**必须是行为测试**——验证用户 / 下游可感知的结果（输入 → 输出 / 副作用），而非内部实现细节。同时必须验证**根本原因**被修复，而不是**症状**消失。遵循《编程领域规则》§6.3。
- **若根因是对象生命周期问题**，优先通过可观察行为捕获；仅在行为层无法稳定捕获时，才用内部断言作为补充诊断，且必须：用 `@pytest.mark.whitebox` 显式标注 + 注明脆弱性 + 仍需附带至少一条行为断言作为主判据。
- **CI 入门资格**：只有行为测试能安全进入 CI 门禁。`whitebox` 标记的内部诊断断言**不得进入 CI 门禁**。判据见《编程领域规则》§9.5。

**小例子**：
> ❌ `assert mock.call_count == 3`——断言调用次数是断言实现细节，重构即脆裂。
> ✅ `assert events["notification_pushed_ts"] <= events["llm_request_ts"]`——断言可观察的时间戳先后。

### 规则 12：鱼骨图分析法

**使用条件**：逐层追问（规则 7）超过 5 层仍找不到明确根因时——线性因果链失效。

**怎么使用**：切换为**多维度并行排查**——不再追问"为什么"，而是从多个独立维度同时扫描。常用维度示例（非穷举，可按项目实际增减）：

| 分类 | 检查内容 |
|------|---------|
| **代码层** | 逻辑错误、边界条件、类型不匹配、异常未捕获 |
| **数据层** | 数据格式变化、缺失字段、编码问题、时序竞争 |
| **环境层** | 依赖版本、环境变量、文件权限、网络连通性 |
| **编排层** | 对象生命周期、插件注册、依赖注入、初始化顺序 |
| **配置层** | 配置文件格式、路径解析、默认值覆盖、合并策略 |

**小例子**：
> 找钥匙，桌上从左到右翻一遍找不到（线性扫描）。换个思路：同时回忆"上次在哪间房、穿哪件衣服、几点回家、谁也在场"（多维度并行）——覆盖面更广。

### 规则 13：时序 bug 的测试设计

**使用条件**：bug 涉及异步、并发、事件顺序、超时等时序问题时。引用《编程领域规则》§6.4 时序测试铁律。

**怎么使用**：
- **禁止仅用零延迟 mock** 验证时序。通过 fake clock、可控 await、显式 sleep 注入可调延迟，覆盖"快路径"和"慢路径"。
- **断言时序不变量**（顺序、间隔、超时边界），而非内部调用次数。
- 关键时序不变量测试标记 `@pytest.mark.timing`，CI 单独调度（见《编程领域规则》§9.4）。
- 时间戳必须来自**同一个受控时钟**，禁止混用真实墙钟。

**小例子**：
> 测"重试间隔 ≥ 退避下限"，不能用零延迟 mock（间隔永远是 0）。要用 fake clock 注入 100ms 延迟，断言间隔 ≥ 退避下限。

## 修复熔断机制

当同一模块的 bug 在完整调试流程中被打回超过 **3 次**仍未解决时，触发熔断：

1. **立即停止调试**，不再尝试修复
2. 将熔断报告写入 `docs/working/circuit_breaker_report.md`，内容包含：
   - 模块名
   - 已尝试的修复方向和失败原因
   - 该模块的已知问题清单
   - 怀疑的架构根因（如职责不清、隐式耦合、状态管理混乱）
3. 调用 task_evaluate 标记熔断状态
4. **禁止继续调试**

---

## 规则

- Bug 修复任务必须先写失败测试再调试修复
- 必须在完整系统中验证修复，不能只依赖隔离测试
- 修复后必须移除所有临时诊断代码
- 同一模块 bug 反复修复超过 3 次仍未解决时，必须触发修复熔断
- 必须先搜现有测试文件，有现成测试能复现 bug 时优先复用，禁止另起炉灶
- 必须添加回归测试防止同类问题复发
- 如果输入包含 state_machine，测试必须覆盖所有状态和转换

---

## 成功标准

- ✅ 原来复现 bug 的失败测试现在通过
- ✅ 全部测试通过（含回归测试）
- ✅ Bug 修复后在完整系统中不再出现
- ✅ 有回归测试防止问题复发
- ✅ 无残留临时诊断代码
