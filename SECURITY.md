# 安全策略

> 本文档说明如何报告 **灵汐 AgentOS** 的安全漏洞，以及我们的处理流程。

---

## 📣 支持的版本

下表列出当前获得安全更新的版本：

| 版本 | 是否支持 |
|------|----------|
| 0.1.x（最新） | ✅ 是 |
| < 0.1.0 | ❌ 否 |

我们仅为最新主版本提供安全修复。发现旧版本漏洞请升级后重测。

---

## 🚨 报告漏洞

**请勿**通过公开 GitHub Issues 报告安全漏洞。

### 推荐渠道（按优先级）

1. **GitHub Security Advisories**（推荐）
   - 访问 https://github.com/jianchen08/Agent-os-open/security/advisories/new
   - 填写漏洞详情（含复现步骤、影响范围、建议修复方案）

2. **邮件**
   - 发送至：`chenjian1306792950@foxmail.com`
   - 邮件主题前缀：`[Security]`
   - 邮件正文使用英文，便于国际安全研究者协作

### 报告应包含

- 漏洞类型（如 SQL 注入、XSS、远程代码执行、SSRF 等）
- 受影响的版本
- 受影响的文件 / 模块路径
- 复现步骤（PoC 代码 / 截图 / 录屏）
- 潜在影响范围
- 是否已公开披露

## ⏱️ 处理流程与 SLA

| 阶段 | 时间 | 我们的动作 |
|------|------|-----------|
| 收到报告 | T+24h | 确认收到，分配跟踪编号 |
| 初步评估 | T+3 天 | 评估严重等级、影响范围、复现可行性 |
| 修复开发 | T+7~30 天 | 取决于严重等级（详见下表） |
| 致谢 / 公开发布 | 修复后 | CVE 申请（如适用）+ Release Notes + 致谢 |

### 严重等级与修复目标

| 等级 | 描述 | 修复目标 |
|------|------|----------|
| 🔴 严重 | 远程代码执行、权限提升、数据泄露 | 7 天内 |
| 🟡 高危 | 认证绕过、敏感信息泄露 | 14 天内 |
| 🟢 中危 | 有限的权限提升、DoS | 30 天内 |
| ⚪ 低危 | 信息泄露、边界场景异常 | 下一版本 |

## 🔐 部署定位与安全最佳实践

> **当前版本的定位**：灵汐 AgentOS 现阶段面向**个人开发者单机自部署**（在 `localhost` 上自己跑、自己用），**不适合多用户共享、不可直接暴露到公网、不具备生产级多租户能力**。原因见下文「认证与授权」。

1. **密钥管理**
   - 所有 API Key 通过 `.env` 文件注入，不要硬编码
   - 即使单机使用也建议轮换默认 `APP_JWT_SECRET_KEY` 和 `DEFAULT_ADMIN_PASSWORD`（默认值 `admin123` 已公开，不换则任何人扫到端口即可登录）
   - `.env` 文件权限设为 `chmod 600`

2. **网络安全**
   - **不要将后端端口 `8988` 直接暴露到公网**；服务默认只监听本机，保持这一默认即可
   - 若确需远程访问，请在前面加反向代理（Nginx / Caddy）并启用 HTTPS + 访问鉴权
   - Redis 端口 `6480` 应仅监听本机

3. **认证与授权**
   - ⚠️ **重要（当前版本限制）**：RBAC 权限矩阵已在 `src/auth/permission_matrix.py` 完成定义，但**路由级强制校验尚未落地**——当前 `require_auth` 仅校验登录态、不校验角色，意味着**任何登录用户均具备管理员权限**（可修改配置、强制触发复盘 `force:true`、删除模型 Provider 等敏感操作）。
   - 正因如此，**当前版本不应用于多用户共享场景**；这也是上方「部署定位」把多用户/公网排除在外的原因。未使用的 IM 通道适配器也应保持关闭。
   - 待路由级 RBAC 强制校验落地后，多用户/生产部署能力才会被正式支持，届时本段定位说明将被更新。

4. **依赖安全**
   - 定期运行 `pip-audit` 和 `npm audit`
   - 关注 GitHub Security Advisories 的告警
   - 启用 Dependabot 自动升级 PR

5. **代码执行隔离**
   - 容器任务默认在 Docker 隔离环境执行（`src/isolation/providers/docker_provider.py`）
   - 不要禁用 `isolation_policy.yaml` 中的安全规则
