# AI辅助编程总纲

## 一、最终目的

**让AI生成的代码，不是"看起来像代码"，而是精确解决需求、架构干净、零冗余、可长期维护的工程资产。**

三层递进目标：

1.  **需求绝对忠诚**：每一行代码都必须能追溯到明确的需求，绝不臆想"潜在需求"。
2.  **架构绝对合规**：严格遵守模块边界、接口契约和分层依赖，禁止跨边界偷跑。
3.  **规范零容忍**：所有工程规范由工具和流程强制执行，不依赖AI的道德自觉。

无论更换什么模型，无论隔多久，产出质量始终稳定、可预期、可审计。

---

## 二、代码的理想状态

-   **极简精准**：没有冗余 `try-catch`、空 `else`、假fallback、`// TODO` 或 `console.log`。
-   **需求可追溯**：每个函数、每个分支都能明确回答"为哪个需求服务"。
-   **架构内聚**：模块内部自由实现，对外只暴露严格定义的公开接口，跨模块调用干净、合法、可检查。
-   **风格绝对一致**：命名、注释、文件组织与项目完全融为一体。
-   **天生安全**：安全性内建于调用链中，而非事后修补。
-   **可直接提交**：人类拿到代码后无需清理、重命名或调整架构，可直接进入 Code Review 或合并。

---

## 三、核心准则

### （一）对编码AI的生成准则

1.  **需求第一，一字不增**：只实现需求文档明确描述的功能。禁止实现任何兼容、扩展、防御性逻辑（除非需求写明）。
2.  **最小输出**：只输出完成任务必需的最少代码，不输出解释、示例或额外帮助。
3.  **严格复用**：优先复用项目内已有函数、类、工具。写新代码前必须先"查阅"现有接口。禁止功能重复的实现。
4.  **尊重私有边界**：绝不访问模块私有的方法、字段或函数，只使用公开接口。
5.  **错误要么处理，要么传播**：不允许吞掉异常。不确定如何处理就直接向上抛，禁止空 `catch` 块和仅日志记录的错误处理。
6.  **放弃假兼容**：不为未来可能的需求留扩展点，不写版本判断、无关数据源切换等无需求依据的逻辑。
7.  **按架构层次思考**：实现前先判断操作属于哪个"为什么"层次（生命周期、消息传递、事件推送），并将代码放入对应模块，禁止跨层调用。

### （二）对审查AI的把关准则

审查AI必须执行以下法定审查步骤，并拥有一票否决权：

1.  **需求追溯审查**：每一行新增代码必须能在需求文档中找到依据。无依据的，标记为"无需求代码"，强制删除。
2.  **架构边界审查（四问）**：
    -   **散点检查**：同一业务概念是否在多个文件中重复出现？若是，驳回并要求统一抽象。
    -   **分叉点检查**：调用方是否因同一操作需要判断多种前置状态？若是，驳回并要求将分叉点封装在模块内部。
    -   **信息泄漏检查**：模块是否知道了另一个模块的实现细节？若是，驳回并强制隔离。
    -   **变化方向检查**：接口是否暴露了易变的实现细节？稳定业务动作是否与易变技术实现耦合？若是，驳回并要求分离。
3.  **冗余与质量审查**：发现"翻译式注释"、无效错误处理、死代码、风格不一致，立即驳回。
4.  **铁面裁决**：只认规则和自动化工具的客观结果，不接受任何"为了好看/稳妥"的主观辩解。

### （三）物理保险准则（自动化强制执行）

这些规则不由AI记忆，而是由工具链在提交/合并时自动执行，违反即失败。

1.  **模块边界物理化**：通过 `no-restricted-imports`、`dependency-cruiser` 等工具，让非法导入直接编译失败。
2.  **架构自动化测试**：编写架构测试（如 `pytest-archon`、`archunit`），检验"某模块不能导入另一模块内部"等约束，CI 必跑。
3.  **需求覆盖率扫描**：自定义脚本检查新增逻辑是否标注了需求ID，无标注则拦截。
4.  **安全与风格 Lint**：ESLint、SonarQube 等工具在提交阶段自动拦截风格、安全、空异常处理等问题。
5.  **冗余模式检测**：配置规则禁止空 `catch` 块、禁止遗留 `console.log`、禁止无 Issue ID 的 `TODO`。

## 四、架构设计方法（定边界的核心方法论）

这套方法论用来回答"模块该怎么划分、接口该怎么设计"，它应该注入到审查AI的检查逻辑和编码AI的思考前置中。

### 重构定边界的四个步骤

| 步骤 | 核心问题 | 动作 |
|------|----------|------|
| 找"散" | 同一个概念在代码里出现了几次？ | 多次出现 → 抽象缺失，需要统一封装 |
| 找"分叉点" | 调用方为了同一个操作需要判断几种情况？ | 调用方有分叉 → 抽象边界错误，应封装在模块内部 |
| 找"谁该知道" | 每个概念，谁需要知道它？ | 不该知道的人知道了 → 边界泄漏，需收回 |
| 找"变化方向" | 什么会变，什么不会变？ | 把"会变的"封装在内部，"不变的"暴露为接口 |

### 抽象边界的四个原则

-   **按"为什么"分层，不按"是什么"分层**：分层应回答"生命周期管理、消息传递、事件推送"等为什么，而非"引擎层、bridge层"。
-   **按"变化频率"定边界**：变化快的实现细节封装在内部，变化慢的业务动作暴露为接口。
-   **按"谁能取消"定权限**：操作不应被拒绝就不设前置条件；操作需要被控制就必须有显式前置步骤。
-   **按"故障隔离"定边界**：如果两个东西的故障会互相影响，它们的边界就定错了。

## 五、这套体系如何运转

```
需求/任务
    │
    ▼
编码AI（生成）       ← 注入"需求第一、严格复用、最小输出"等准则
    │
    ▼
物理保险（自动拦截） ← 架构测试、Lint、依赖检查、需求覆盖扫描
    │ (通过)
    ▼
审查AI（把关）       ← 执行"需求追溯+架构四问+冗余审查"
    │ (通过)
    ▼
代码仓库（干净资产）
```

## 六、最终效果

-   AI不再是热衷展示"想得周全"的新手，而是对需求保持绝对忠诚、一字不多的冷酷执行者。
-   生成的代码自带清晰边界，概念不重复，分叉点封装在内，修改一处不扩散全局。
-   每一次输出都像经历了资深工程师的重构，干净、精确、可追溯。
-   代码即架构，架构即文档，团队协作不再需要"洗稿"AI代码，可直接信任并合并。

**这就是AI编程要达到的终极状态：不是写得快，而是写得对，且每一次都对。**
