## 修复目标

修复"触发器通知（system_notification）刷新后渲染两次"。建立与 AI 消息同款的 **id 契约**，让流式 system 气泡和后端落库的 system 记录共享同一个 record_id，前端现有 id 去重（`isCoveredByApi`）即可自然处理，无需新增去重逻辑。

## 根因（已验证）

对比 AI 消息与 system 通知，差异在 **id 契约**，不是 sequence：

- **AI 消息不重复的原因**：bridge `emit_start` 生成 hex `message_id`，写入 `state["preset_ai_record_id"]`（bridge_core.py:242-244）；track AI 记录用它作 `record_id`（plugin.py:370-375）；前端流式气泡 id = 该 `message_id`。刷新后 API 返回的 AI 记录 record_id == 气泡 id → 前端按 id 去重成功。
- **system 通知重复的原因**：前端流式气泡 id = `sys_<前端UUID>`（lifecycleHandlers.ts:130，前端随机生成）；track system 记录**未设 record_id**（plugin.py:337-346 省略），存储自动生成随机 uuid（execution_record_storage.py:523-524）。两者 id 不同、seq 不同（事件 seq=K，record seq=K+1，两次独立 `entry.next_sequence()`）→ `isCoveredByApi` 和 `role::seq` 指纹都匹配失败 → 重复。
- **docstring/code 漂移**：consume_pending_notifications docstring（engine_iteration.py:169-171）声称"system 不落库避免二次渲染"，但实际代码 L250-256 写了 user_input，触发 track 落 system 记录。

## 改动清单

### 后端：给 system 通知建立 id 契约（模仿 AI 的 preset_ai_record_id）

**1. `src/pipeline/engine_iteration.py` consume_pending_notifications（~L275-279）**

在循环推送 system 通知时，让 `emit_notification` 返回 record_id，并把本轮 system 通知的 record_id 写入 state 供 track 复用。

由于多条 system 通知在 consume 里会合并（L242 `"\n\n".join`）后写 user_input，track 也只落一条合并记录——所以 id 契约按"本轮 system 通知用一个 record_id"建模（与"一轮一个 AI record_id"对称）。单条通知场景（如触发器）天然一对一。

具体：`emit_notification` 增加 `record_id` 形参；consume 为本轮 system 通知生成一个 hex id，逐条事件带上它，同时写 `state["_pending_system_record_id"]`（类似 `preset_ai_record_id`）。若本轮有多条 system，用第一条的 id（合并记录场景下一致即可）。

**2. `src/pipeline/bridge_core.py` emit_notification（L475-521）**

- 接受 `record_id` 形参（缺省时内部生成 hex，保证向后兼容）。
- 把 record_id 放进事件 payload（字段名 `record_id`，与 stream_start 的 message_id 命名区分但语义平行），前端据此设消息 id。
- 保留现有 `notification_id`（前端内存去重仍用）。

**3. `src/plugins/output/track/plugin.py` system 记录（~L337-346）**

读 `ctx.state.get("_pending_system_record_id")`，作为 `injected_record` 的 `record_id=`。读不到则保持现状（随机 uuid，向后兼容）。

### 前端：用后端 record_id 作为 system 消息 id

**4. `src/services/websocket/streaming/lifecycleHandlers.ts` handleSystemNotification（~L129-153）**

- 优先用事件里的 `record_id`（`data?.record_id`）作为消息 `id`；缺失时 fallback 到现有的 `sys_${generateUUID()}`。
- 这样流式气泡 id == 后端 record_id，刷新后 API 返回同 record_id → `isCoveredByApi` 按 id 自动去重，无需新逻辑。

**5. （可选，向后兼容）`src/channels/api/routes_threads.py` system 记录映射（L551-560）**

已确认此处默认用 `record.record_id` 作消息 id，无需改动。仅在注释里点明"system record_id 现与 WS 事件一致"。
