# 修复计划：输入校验缺口（AC 静默吞异常 + file_write 漏传 action）

## 核心原则
你定的逻辑：**空（不填）→ 可以，提交成功；填了 → 必须填对，填错就拒绝提交。**
两个问题同一个修法模式：把"静默吞掉/含糊错误"改成"拒绝 + 明确反馈 LLM"，与现有 `INVALID_METRIC_ID`（task_submit/tool.py:574-583）模式完全一致。

---

## 修复 1：AC 静默吞异常 → 拒绝提交

**文件**：`src/tools/builtin/task_submit/tool.py`

### 改动 A：execute 入口（第 540-545 行）
现状：`acceptance_criteria` 非 dict 时，`logger.warning` + `acceptance = {}` 静默重置，继续提交。
改为：非 dict（含 list/string 等）时，返回 `create_failure_result`，`error_code="INVALID_ACCEPTANCE_CRITERIA"`，错误信息明确告知"必须是 dict（key=指标ID，value=配置对象），传了 list/字符串等错误类型"。

位置：放在 description 长度校验（532-538）之后、metric_id 校验（558-589）之前，保持现有校验链顺序。

注意保留第 547-556 行"不 fallback/不默认/不覆盖"的注释铁律，它与新逻辑一致（空 dict 时什么都不做，正常往下走提交）。

### 改动 B：_build_metadata 第二道防线（第 1480-1488 行）
现状：非 dict 时 `logger.warning` + 跳过存储（静默丢失）。
改为：加 `assert isinstance(acceptance_criteria, dict)`——既然 execute 入口已拦截非 dict，这里不应再出现非 dict；出现即编程错误，让断言报错而非静默吞。
（这是防御性收口，遵循"禁止空 catch/仅日志记录"铁律。）

## 修复 2：file_write 漏传 action → 明确错误

**文件**：`src/tools/builtin/file_write/tool.py`

### 改动 C：execute 内部分发（第 255-270 行）
现状：`action = inputs.get("action")` 无默认值；漏传→None→落到兜底→"不支持的操作: None"（含糊）。
改为：区分两种情况，都给 LLM 明确反馈：
- action 缺失（None）→ `error_code="MISSING_ACTION"`，错误信息"必须传 action 参数，可选值：write/search_replace/insert/delete_lines/append"
- action 非法（不在枚举内）→ `error_code="INVALID_ACTION"`，错误信息列出传入值 + 合法枚举

不改 schema（action 已在 required 里，本就该必填），不加默认值（默认成 write 会有静默覆盖风险），不动 tool_core 的 schema 校验绕过问题（架构改动面大，超出本次范围）。
