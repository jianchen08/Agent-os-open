# 实现思维原则

每次拿到实现需求，动手写代码之前先过这三步。

## 1. 用一句话说本质

不看代码，先用大白话说清楚需求到底要什么。说不清就是没理解，停下来想清楚再动手。

错误示范：「在 workspace_lifecycle 的 worktree 创建流程里加一个继承分支，复制旧文件到新目录」
正确示范：「新任务要能看到旧任务的文件 → 把旧路径直接给它」

## 2. 问三个问题选方案

- **能不能不做？** — 需求真的需要新逻辑吗，现有机制换个用法能不能解决？
- **能不能少做？** — 最少改几行代码就能实现？先写最小方案，复杂度后面按需加。
- **能不能绕过？** — 在流程入口/出口直接短路，而不是在流程中间插逻辑。

## 3. 警惕三个思维陷阱

### 叠加思维
在既有复杂流程里改东西时，习惯性往上面加层（加参数、加分支、加 fallback），而不是退一步看整个流程能不能跳过。
- 表现：改了一处不够又补一处，再补一处 → 方向可能错了
- 表现：觉得"既然流程里已经有一套机制了，应该复用它" → 可能根本不需要走这套机制

### 抽象绑架
代码里已有的抽象（mode、factory、lifecycle）让人默认"新功能也得是其中一种"。
- 表现：把新需求硬塞进已有的 enum/分类体系
- 表现：花大量时间想"该归到哪个类别"而不是"需不需要分类"

### 锚定效应
第一个想到的方案持续影响后续判断。一旦开始在某条路上写代码，后续决策都基于这条路。
- 表现：被指出问题后，仍在原来框架里修修补补
- 表现：每次修补都觉得自己"快完成了"，其实方向偏了

## 自检信号

连续补了 3 个补丁还没彻底解决 → 停下来重新审视方案，而不是继续打补丁。
听到「不用 xxx 啊」时 → 不是在说某个步骤可以省略，是说整个思路要换。

## 4. 架构设计方法论（总纲§四，定边界的核心方法论）

这套方法论用来回答"模块该怎么划分、接口该怎么设计"。编码时用于前置思考，审查时用于检查逻辑。

### 重构定边界的四个步骤

| 步骤 | 核心问题 | 动作 |
|------|----------|------|
| 找"散" | 同一个概念在代码里出现了几次？ | 多次出现 → 抽象缺失，需要统一封装 |
| 找"分叉点" | 调用方为了同一个操作需要判断几种情况？ | 调用方有分叉 → 抽象边界错误，应封装在模块内部 |
| 找"谁该知道" | 每个概念，谁需要知道它？ | 不该知道的人知道了 → 边界泄漏，需收回 |
| 找"变化方向" | 什么会变，什么不会变？ | 把"会变的"封装在内部，"不变的"暴露为接口 |

### 抽象边界的四个原则

- **按"为什么"分层，不按"是什么"分层**：分层应回答"生命周期管理、消息传递、事件推送"等为什么，而非"引擎层、bridge层"。
- **按"变化频率"定边界**：变化快的实现细节封装在内部，变化慢的业务动作暴露为接口。
- **按"谁能取消"定权限**：操作不应被拒绝就不设前置条件；操作需要被控制就必须有显式前置步骤。
- **按"故障隔离"定边界**：如果两个东西的故障会互相影响，它们的边界就定错了。
