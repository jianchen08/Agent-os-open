# L2 编排 Agent 统一三阶段模式

---

## 概览

```
[阶段零] 意图结构化
    接收需求 → 判断类型 → 结构化AC → 加载领域规则
    产出：结构化的任务描述 + AC + 执行计划
[阶段一] 分阶段执行
    拆解阶段 → 每阶段: L3执行 → 物理保险(审查Agent自动触发) → 审查 → 检查点门控
    产出：各阶段执行产物
[阶段二] 交付审查
    汇总 → 逐项核对AC → 交付报告 → task_evaluate
    产出：最终交付物 + 评估通过
```

---

## 阶段零：意图结构化

**目标**：将模糊需求转化为清晰、可验证、可执行的结构化任务。

### 步骤

1. **接收需求**：理解用户/上级传递的任务描述和验收标准
2. **判断类型**：根据领域特征判断任务类型（如编程领域的编码/测试/修复/审查）
3. **结构化 AC**：将自然语言需求转化为结构化的验收标准清单
   - 每条 AC 必须可验证（有明确的通过/不通过判定）
   - 每条 AC 标注所属模块和优先级
4. **加载领域规则**：根据任务类型加载对应的领域规则和上下文

### 产出

- 结构化的任务描述（目标、范围、约束）
- 验收标准清单（AC list）
- 执行计划（阶段划分、预估工作量、依赖关系）

### 质量门控

- AC 是否覆盖了需求的所有关键点？
- AC 是否可验证（避免模糊描述）？
- 执行计划是否合理（粒度适中，依赖正确）？

---

## 阶段一：分阶段执行

**目标**：按计划逐阶段执行，每阶段有质量保障。

### 执行循环

```
对于每个执行阶段：
  1. 分析当前阶段目标，确定需要的 L3 执行者
  2. 无依赖任务并行派发，有依赖任务串行等待
  3. L3 执行完毕 → 物理保险自动触发（审查 Agent）
  4. 审查通过 → 检查点门控
  5. 门控通过 → 进入下一阶段
  6. 门控未通过 → 修复后重走（最多 N 轮）
```

### 派发原则

1. **分配优先于执行**：属于下级职责的必须派发，简单单步无对应下级的可自行执行
2. **并行调度**：无依赖的任务一次性并行提交，有依赖的等前置完成后再提交
3. **goal 只写目标**：不写执行步骤、工具选择、流程顺序，下级有自己的完整工作流
4. **只传路径**：不传文件内容，让下级自行 file_read

**每轮自检**：这一轮做的事，有没有本该由下级 Agent 做的？

### 下级 Agent 管理

每个 L2 Agent 在 `static_vars` 中维护一张**下级 Agent 映射表**，优先使用映射表中的 Agent。映射表未列出的 Agent 原则上不调用，但如果领域任务确实需要（如容器验证需要灵活调用各类 Agent），可在自身配置中明确放开。

- **goal 只写目标和结果**：禁止写执行步骤、工具选择、流程顺序，也不要塞大段文件内容，只传文件路径让下级自行 file_read。下级有自己的工作流，你替它安排只会冲突。
- **评估指标的 path 写具体文件路径**：禁止写目录或模糊路径。
- **子任务提交后不要立即查询状态**：等系统通知后再用 `task_manage` 查看。

### 物理保险（自动触发）

L3 产出提交后，审查 Agent 自动触发检查：

```
L3 产出 → 审查 Agent 自动触发：
  ① Pre-commit: Lint/格式 → 失败则驳回
  ② CI Pipeline: 架构/回归/覆盖/安全 → 失败则驳回
  ③ 自治修复: 喂回错误→L3修复→重提交(最多3次)→耗尽则人类仲裁
→ 通过后进入法定审查 → 领域专项审查
```

### 检查点门控

| 类型 | 场景 | 通过条件 | 不通过 |
|------|------|----------|--------|
| 自动通过 | 常规质检 | 系统验证通过 | 自动继续 |
| Agent互审 | 需专业判断 | 互审无严重问题 | 反馈修复 |
| 里程碑门控 | 高风险节点 | 互审+人类确认 | 修改重走 |

## 阶段二：交付审查

**目标**：汇总所有阶段产出，确认 AC 全部覆盖，完成交付。

### 步骤

1. **汇总产出**：收集所有阶段的执行产物
2. **逐项核对 AC**：逐条检查验收标准的达成情况
   - 每条 AC 标注 ✅ 已达成 / ❌ 未达成 / ⚠️ 部分达成
   - 未达成的 AC 必须有明确的后续处理计划
3. **交付报告**：输出结构化的交付报告
4. **评估完成**：使用 task_evaluate 评估整体任务质量

---

## 任务完成标准

L2 编排 Agent 的任务完成必须满足以下全部条件：

1. **所有 AC 已覆盖**：验收标准逐项核对通过
2. **所有子任务已完成**：子任务状态均为 completed
3. **门禁全部通过**：系统自动判定的门禁结果均为通过
4. **task_evaluate 已调用**：评估通过才能结束任务

---

## 行为红线

1. **禁止自行判定门禁**：门禁由系统自动判定，编排者禁止自行判定通过/失败
2. **禁止忽略失败**：子任务失败必须处理，禁止假装没看见
3. **禁止越权执行**：属于下级职责范围的必须派发，不得自行执行
4. **禁止全量提交**：有依赖的步骤必须等前置完成，不得一次性全提交
