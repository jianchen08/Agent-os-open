# 需求结构化流程规范

---

## 1. 结构化流程

### 1.1 总体流程

```
原始需求 → 功能点提取 → 模块归属分析 → 验收标准分解(按模块分组) → 状态机识别 → 边界场景补充 → 自检 → 传递
```

不可跳过任何步骤。信息不足时暂停并追问用户，而非猜测。

### 1.2 步骤一：功能点提取

**目标**：从自然语言需求中提取所有功能点。

**方法**：

1. 找出所有"动词+宾语"组合，每个组合是一个功能点
2. 找出所有条件分叉（"如果…则…"），每个分叉生成独立功能点
3. 找出所有隐含行为（"提交后"隐含 loading 状态，"失败后"隐含 error 处理）

### 1.3 步骤二：模块归属分析

**目标**：将功能点分配到模块，明确模块边界和跨模块接口。

**分析方法**：

1. **识别模块**：根据功能点的职责领域，将功能点归类到对应模块。同一职责领域的功能点属于同一模块
2. **标记模块边界**：对每个模块，区分：
   - **内部功能**：只在模块内部使用的功能，不对外暴露
   - **对外功能**：需要被其他模块调用的功能，即模块的公共接口
3. **识别跨模块依赖**：当功能点 A 依赖功能点 B，且它们属于不同模块时，标记为跨模块依赖，明确：
   - A 调用 B 的哪个公共接口
   - 接口的输入输出是什么
   - 依赖方向（A→B），确保不产生循环依赖
4. **按模块分组 AC**：后续步骤产出的验收标准按模块分组传递，确保每个 code_writer 任务只关注一个模块

**输出格式**：

```
模块A：
  - 对外接口：[功能点1, 功能点2]
  - 内部实现：[功能点3, 功能点4]
  - 依赖：无

模块B：
  - 对外接口：[功能点5]
  - 内部实现：[功能点6]
  - 依赖：模块A.功能点1（调用其公共接口）
```

### 1.4 步骤三：验收标准分解

**目标**：将每个功能点转化为可二值判断的验收标准（AC）。

**分解原则**：

- 每个 AC 必须能回答"通过/未通过"，不能有模糊空间
- 每个 AC 必须描述**可见效果**（用户能看到什么/系统状态变成什么）
- 每个 AC 标注优先级：must（必须实现）/ should（应该实现）/ nice（锦上添花）

**分解维度**（至少从以下角度思考）：

| 维度 | 思考问题 |
| ---- | -------- |
| 触发条件 | 什么操作/事件触发这个行为？ |
| 预期行为 | 触发后系统应该做什么？ |
| 可见反馈 | 用户能看到什么变化？ |
| 约束条件 | 有无数量、时间、权限等限制？ |
| 异常路径 | 如果失败/出错应该怎样？ |

### 1.5 步骤四：状态机识别

**目标**：识别所有涉及状态变化的组件，建立状态转换关系。

**识别方法**：

1. 找出所有有"异步操作"的组件（数据获取、表单提交、文件上传等）
2. 对每个组件，列出其所有可能的状态
3. 对每两个状态之间的转换，描述：触发事件、前置条件、转换后的可见效果

**必须覆盖的状态维度**：

- 初始态（idle）
- 进行中（loading/submitting/processing）
- 成功（success）
- 失败（error）
- 空数据（empty）—— 对有数据展示的组件
- 特殊态（如锁定、禁用、过期等业务特定状态）

### 1.6 步骤五：边界场景补充

**目标**：识别需求中未明说但可能发生的边界和异常场景。

**思考维度**：

| 数据维度 | 思考方向 |
| -------- | -------- |
| 集合类 | 空集、单元素、大量数据 |
| 数值类 | 零、负数、边界值、超范围 |
| 字符串类 | 空串、超长、特殊字符 |
| 时间类 | 过去、当前、未来 |
| 网络类 | 超时、断连、慢网络 |
| 并发类 | 重复提交、快速连续操作 |

对每个识别出的边界场景，记录：触发条件、预期行为。

### 1.7 步骤六：自检

结构化完成后，逐项自检：

1. 每个 AC 是否都能二值判断？
2. 每个 AC 是否描述了可见效果？
3. 状态机中的状态是否覆盖了 AC 中涉及的所有状态？
4. 状态转换是否有遗漏？
5. 边界场景是否从至少 3 个维度进行了思考？
6. 是否有需求信息不足导致无法判断的 AC？
7. **每个模块是否有清晰的对外接口和内部实现的区分？**
8. **跨模块依赖是否都是单向的（无循环依赖）？**
9. **AC 是否按模块分组，每组 AC 可以独立交给一个 code_writer 任务？**

### 1.8 步骤七：传递

将结构化结果传递给下游。**本阶段产出的实现级 AC 不是凭空新建，而是对方案文档 §四
方案级 AC 的细化，每条实现级 AC 必须标注 `traces_to: AC-{方案级编号}`，建立追溯。**
方案级 AC 未覆盖的实现细节，归到所属任务级 AC 下，不得新增方案未承诺的功能。

1. 验收标准清单（按模块分组，标注 traces_to）→ code_writer（每个任务只负责一个模块）
2. 模块间接口定义 → code_writer（作为编码约束）
3. 状态机定义 → test_debug_agent（作为测试设计输入）
4. 边界场景清单 → test_debug_agent（作为异常测试输入）
5. **测试命名锚定**：每条实现级 AC 至少有一个以 AC-ID 命名的测试（如 `test_AC1_create_coupon`），
   使状态矩阵能从测试结果自动推导（确定性，不靠手动盖章）

---

## 2. 信息不足时的处理流程

当原始需求过于模糊，无法完成结构化时：

1. 标记为"信息不足"，列出缺失的信息点
2. 通过 human_interaction 向用户追问具体问题
3. 追问后补充结构化内容
4. **禁止猜测**——宁可追问也不能编造 AC

---

## 3. 格式灵活性

本规范不约束结构化输出的具体格式（YAML/JSON/Markdown），
各 Agent 可根据自身需要选择最适合的表达方式。

**核心要求是内容完整性**：验收标准、状态机、边界场景三个部分缺一不可。
