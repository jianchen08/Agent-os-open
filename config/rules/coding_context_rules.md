# 编程场景核心上下文规则

---

## 信息获取顺序（5级）

### 层级定义

| 优先级 | 层级 | 获取什么信息 | 用什么工具 | 为什么 |
|--------|------|-------------|-----------|--------|
| 1 | **项目级** | 系统架构、API设计、配置文件、项目结构 | `file_read` 读取 `.project/` 目录 | 全局性修改、跨模块开发必须先理解整体架构 |
| 2 | **模块级** | 模块需求、逻辑结构、接口定义、模块依赖 | `file_read` 读取 `{模块}/{模块}.md` | 模块内开发需理解模块边界和对外接口 |
| 3 | **代码级** | 具体组件源码、函数实现、类型定义 | `file_read` 或 `enhanced_search` | 单组件开发需理解现有实现细节 |
| 4 | **规范级** | 代码风格、类型注解、命名约定 | `file_read` 读取 `config/rules/` 目录 | 确保代码符合团队编码规范 |
| 5 | **依赖级** | 第三方库文档、API参考、工具说明 | `fetch` 获取官方文档或 `memory` 检索 | 解决技术问题、正确使用依赖库 |

### 工具使用说明

| 工具 | 适用层级 | 使用场景 |
|------|---------|---------|
| `file_read` | 项目级、模块级、规范级 | 读取 Markdown 文档和源码文件 |
| `enhanced_search` | 代码级 | 搜索函数、类、变量的定义位置 |
| `fetch` | 依赖级 | 获取外部库的官方文档 |
| `memory` | 依赖级 | 检索已存储的领域知识 |

### 层级递进规则

```
1. 项目级文档 → 2. 模块级文档 → 3. 代码级源码 → 4. 规范级规则 → 5. 依赖级文档
     ↓                ↓                ↓               ↓              ↓
   理解架构        理解模块边界      理解实现细节     遵循编码规范     正确使用依赖
```

**强制规则**：必须按顺序递进，禁止跳级获取上下文。确需跨级时，应先完成中间层级的信息获取。

## 上下文范围（5种任务类型）

### 规则概述

根据任务类型不同，需要读取的上下文范围有所不同。本节定义每种任务类型的必需上下文和可选上下文。

### 任务类型定义

#### 1. 新模块开发

| 上下文类型 | 必需读取 | 说明 |
|------------|---------|------|
| 项目级 | ✅ | `.project/charter.md`（架构）、`.project/` 目录 |
| 模块级 | ✅ | 新模块的 `{模块}.md`（需新建） |
| 组件级 | ✅ | 组件级文档模板 |
| 规范级 | ✅ | `config/rules/` 下所有规则 |
| 依赖级 | ⬜ 可选 | 技术栈官方文档 |

#### 2. 现有模块修改

| 上下文类型 | 必需读取 | 说明 |
|------------|---------|------|
| 项目级 | ⬜ 跨模块时 | 相关项目级文档 |
| 模块级 | ✅ | 模块的 `{模块}.md` |
| 组件级 | ✅ | 待修改组件的 `{组件}.md` |
| 代码级 | ✅ | 待修改的源码文件 |
| 规范级 | ✅ | 相关规范文件 |

#### 3. Bug修复

| 上下文类型 | 必需读取 | 说明 |
|------------|---------|------|
| 错误日志 | ✅ | 错误堆栈、错误信息 |
| 代码级 | ✅ | 报错位置的源码 |
| 测试用例 | ✅ | 相关的测试文件 |
| 模块级 | ⬜ 可选 | 上下文复杂时 |
| 规范级 | ⬜ 可选 | 规范不明确时 |

#### 4. 重构优化

| 上下文类型 | 必需读取 | 说明 |
|------------|---------|------|
| 模块级 | ✅ | 模块的 `{模块}.md` |
| 组件级 | ✅ | 待重构组件的文档 |
| 代码级 | ✅ | 现有源码 |
| 测试覆盖 | ✅ | 测试文件 |
| 规范级 | ✅ | 相关规范 |

#### 5. 测试编写

| 上下文类型 | 必需读取 | 说明 |
|------------|---------|------|
| 代码级 | ✅ | 被测代码源码 |
| 接口定义 | ✅ | 函数/类签名、API接口 |
| 测试规范 | ✅ | `config/rules/testing_rules.md` |
| 示例测试 | ⬜ 可选 | 项目已有测试文件 |
| Mock策略 | ⬜ 可选 | `testing_rules.md` Mock章节 |

### 上下文选择规则

```
任务类型 → 确定必需上下文 → 按优先级读取 → 补充可选上下文 → 开始编码
```

**强制规则**：
1. 必需上下文必须全部读取方可开始编码
2. 可选上下文根据任务复杂度酌情补充
3. Bug修复优先读取错误日志和报错代码

## 先读后写规则（5条）

### 规则概述

编码前必须先读取相关上下文，遵循「先读后写」原则，确保编码决策有据可依。

### 规则列表

#### 规则 1：确定模块归属

```
编码任务 → 确定模块归属 → 读取模块级文档 → 理解模块边界 → 再动手编码
```

**具体要求**：
1. 编码前必须明确目标模块
2. 读取模块级文档（`{模块}/{模块}.md`）
3. 确认模块边界和对外接口
4. 跨模块修改需同时读取相关模块文档

#### 规则 2：按索引定位

```
查找内容 → 定位文档 → 读取「结构」章节 → 通过索引定位 → 不盲目搜索
```

**具体要求**：
1. 每层文档通过「结构」章节索引入口
2. 禁止盲目使用 `enhanced_search` 搜索全项目
3. 搜索前先确认目标文档的「结构」章节

#### 规则 3：新增文件更新清单

```
新增文件 → 立即更新「结构 > 文件清单」→ 保持文档与代码同步
```

**具体要求**：
1. 新增任何文件必须更新所在模块的「文件清单」
2. 更新时机：编码完成后立即更新文档
3. 清单格式：`{文件名} - {文件用途} - {创建时间}`

#### 规则 4：新增组件创建文档

```
新增组件 → 创建组件级文档 → 包含需求+逻辑+结构 → 再编码
```

**具体要求**：
1. 新增组件必须创建对应文档
2. 文档必须包含：需求说明、逻辑说明、结构说明
3. 文档模板参考 `docs/templates/component_template.md`

#### 规则 5：修改前确认测试覆盖

```
修改代码 → 检查测试覆盖 → 无覆盖则先补充测试 → 有覆盖再修改
```

**具体要求**：
1. 修改现有代码前先确认是否有测试覆盖
2. 无测试覆盖的代码修改属于高风险操作
3. 优先补充测试，再进行代码修改
4. 重构必须先确认测试覆盖
