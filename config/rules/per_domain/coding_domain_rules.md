# 编程领域规则

---

## 一、核心原则

### 1.1 需求第一，一字不增

- 只实现需求文档明确描述的功能
- 区分"不言自明的系统需求"（操作连贯性、状态反馈、安全底线）与"有争议的体验假设"（用户没说的一律不做）
- 禁止任何"用户可能想…"的假设

> 小例子：需求说"用户能登录"，就只做登录。不要顺手加"记住我""第三方登录""登录后推荐内容"——这些用户没说。

### 1.2 最小输出与优先修改

- 在满足需求的前提下，第一优先级永远是减少/修改现有代码，而非添加新代码
- 只输出必需的最少代码，禁止冗余（多余 try-catch、空 else、假 fallback、TODO）

> 小例子：已有 `formatDate()` 工具函数，需求要"显示日期"——调用它，而不是新写一个 `formatDateV2()`。

### 1.3 严格复用与边界尊重

- 写新代码前先查阅现有接口，优先复用
- 绝不访问模块私有成员，只使用公开接口
- 遇阻碍（如缺少参数）必须先沟通，禁止静默绕过或重写已有API

> 小例子：某个内部方法 `_validate()` 缺个参数，正确做法是找作者沟通加参数或开放接口，不是自己复制一份重写。

### 1.4 错误处理铁律

- 要么处理要么传播，禁止吞异常
- 禁止空 catch 块或仅日志记录的错误处理
- 错误是值，应可恢复，边界处翻译错误，日志与用户提示分离

> 小例子：`except Exception: pass` 是吞异常——要么 retry / 给默认值（处理），要么 `raise`（传播），不能假装没发生。

### 1.5 Bug修复安全

- **止血不截肢**：修复Bug必须定位并修改精确错误，严禁退回旧方案或绕开
- **重构时须注入模块需求清单**，确保外部行为不变

> 小例子：新登录方式有 bug，正确做法是修新方式的精确错误；不能图省事退回旧登录方式——那是绕开，不是修复。

### 1.6 复杂行为建模原则 [error]

**使用条件**：涉及**异步、并发、时序、状态机**等复杂行为时——禁止直接改代码，必须先定义行为模型。

**怎么使用**：
- 行为模型必须以**测试用例**形式表达，而非自然语言描述或图示（见 §6.3）。
- "理想时序图"等画图手段只是中间思维工具，不作为强制产物；规则强制要求的是"先写行为测试"。
- 没有可执行的行为测试之前，不允许提交针对该复杂行为的实现改动。

**小例子**：
> 重试逻辑有 bug，不要直接改重试代码。先写一个测试断言"重试间隔 ≥ 退避下限"，跑红灯，再改代码让它变绿。

## 二、SOLID 原则

### 2.1 SRP 单一职责 [error]

每个类或模块只有一个变更原因。

### 2.2 OCP 开闭原则 [warning]

对扩展开放、对修改关闭。

### 2.3 LSP 里氏替换 [warning]

子类型必须可以替换其基类型。

### 2.4 ISP 接口隔离 [warning]

客户端不应依赖它不需要的接口。

### 2.5 DIP 依赖倒置 [error]

依赖抽象而非具体实现。

## 三、兼容性/Fallback 原则

> **一句话决策**：内部代码（双方都在你控制下）直接改，不降级不兼容；只有**外部依赖**（第三方 API、不可控客户端、已发布接口）才需要兼容/降级方案。

### 3.1 内部 vs 外部——是否需要兼容/降级 [error]

**使用条件**：接口变更、或功能可能在外部故障下不可用时。

**怎么使用**：
- **内部模块**（同一仓库、可控微服务）：直接改，不保留旧接口、不设计降级。
- **外部依赖**（第三方 API、已发布接口、不可控客户端）：才需要兼容/降级方案。
- AI 必须能通过代码搜索确认"接口所有消费方是否都在可分析范围内"——能全量感知就按内部处理，不能就按外部处理。

**小例子**：
> 改内部 `UserService` 的方法签名：所有调用方都在仓库里，直接改 + 同步更新所有调用点，不保留旧方法。
> 改对外发布的 API 字段：可能有第三方在用，必须保留旧字段 + 新字段给默认值。

### 3.2 兼容策略——外部依赖的字段变更 [error]

**使用条件**：需要兼容外部依赖（调用方不可全量感知）时。

**怎么使用**：
- **新增是安全的，删除/修改是危险的**：字段只能增不能删，废弃字段保留默认值。
- 新代码不能假设老调用方会传新字段（必须有默认值）；老代码不能假设新调用方会传老字段（必须容忍缺失）。
- 有计划地淘汰而非突然断裂：Deprecate（标记废弃）→ Warn（警告期）→ Remove（移除），每阶段至少经历一个版本。
- 所有外部输入严格校验——兼容性不等于放任。

**小例子**：
> API 原有 `name` 字段，现在要拆成 `firstName`/`lastName`：保留 `name`（标记 deprecated），新增 `firstName`/`lastName` 给默认值，下个大版本再移除 `name`。

### 3.3 降级策略——外部故障时怎么办 [error]

**使用条件**：功能依赖外部服务，且该服务可能故障时。

**怎么使用**：
- **功能完整性优先**：任何降级方案不得导致核心功能缺失。兼容是"用不同方式达到相同结果"，不是"得到残缺结果"。
- **渐进降级**：方案 A（最优）失败 → 方案 B（次优）→ 方案 C（兜底）。路径必须明确，不能靠运气。最终兜底宁可抛明确错误，也不能假装一切正常。
- **用户必须有感知**，禁止静默降级：核心流程（数据一致性、业务提交）→ 阻塞并明确报错；非核心流程 → 不阻塞但必须给降级提示。

**小例子**：
> 支付接口故障：核心流程，必须阻塞 + 明确报错"支付暂不可用"，不能静默跳过。
> 头像 CDN 故障：非核心，可降级显示默认头像 + 提示"头像加载失败"，但不能让整个页面崩。

### 3.4 优先级链条 [error]

当各目标冲突时，按此顺序取舍：

1. 功能完整性（不可妥协）
2. 数据正确性（不可妥协）
3. 用户体验完整性（不可妥协）
4. 兼容性（仅限外部依赖，尽量满足）
5. 性能（可接受轻微下降）
6. 代码优雅度（最先牺牲）

## 四、编码规范

### 4.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量/函数 | camelCase（JS/TS）或 snake_case（Python） | `getUserInfo`, `get_user_info` |
| 类/接口 | PascalCase | `UserService`, `HttpClient` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 私有成员 | _leadingUnderscore | `_internalState` |
| 布尔变量 | is/has/can/should 前缀 | `isActive`, `hasPermission` |

### 4.2 代码风格

- **函数式优先**：优先纯函数，避免副作用
- **按职责拆分，不按行数**：一个文件出现 ≥2 个不相干的变更原因（职责耦合）时才拆分，行数不是判据。AI 主导维护的项目单文件软参考线放宽到约 1000 行（本项目实测约 3% 文件超千行且均为核心引擎，未成为维护负担）
- **量化阈值**（圈复杂度/函数行数/参数个数）：见 §九.3 唯一真相源

### 4.3 TypeScript 规范

- 严格模式（strict: true）
- 禁用 any（除非有明确注释说明原因）
- 接口优于 type（除非需要联合类型/交叉类型）
- 启用 noImplicitAny、strictNullChecks

### 4.4 Python 规范

- 遵循 PEP 8
- 类型注解：具体规范见 python_type_hints.md
- docstring：公共函数/类必须有 docstring
- import 顺序：标准库 → 第三方 → 本地

### 4.5 Go 规范

- 命名：导出用 PascalCase，私有用 camelCase，包名简短全小写
- 错误处理：`if err != nil` 显式处理，不用 panic 处理常规错误
- 接口：小而专（1-3 个方法），由消费方定义而非实现方
- 格式化：强制 `gofmt` + `goimports`
- 测试：`_test.go` 与源文件同包，使用标准 `testing` 包

### 4.6 React / 前端规范

- 组件命名 PascalCase，文件名与组件名一致（`UserCard.tsx`）
- Hooks 命名强制 `use` 前缀（`useUserList`）
- 不可变状态：永远不直接 mutate state，用展开运算符/解构创建新对象
- 事件处理：`handleClick`（类组件）/ `onClick` 接收（函数组件）
- 测试：Jest + React Testing Library，测试用户行为而非实现细节

## 五、文件组织

### 5.1 目录结构约定

```
src/
├── modules/          # 业务模块（按功能域划分）
│   ├── auth/         # 认证模块
│   └── user/         # 用户模块
├── shared/           # 共享代码（跨模块复用）
│   ├── utils/        # 工具函数
│   └── types/        # 共享类型定义
├── infrastructure/   # 基础设施层
└── config/           # 配置文件
tests/
├── unit/             # 单元测试（镜像src结构）
├── integration/      # 集成测试
└── fixtures/         # 测试固件
```

### 5.2 模块边界

- 每个模块有清晰的公共接口（index.ts / __init__.py）
- 模块间通过公共接口通信，禁止跨模块访问内部文件
- 共享代码放 shared/，业务代码放 modules/
- 禁止循环依赖

## 六、测试要求

### 6.1 覆盖率标准

具体标准见 testing_rules.md

### 6.2 测试编写规范

测试编写规范（AAA 模式、命名、Mock 策略）详见 testing_rules.md。

### 6.3 行为测试原则 [error]

**使用条件**：编写任何测试时——决定断言什么、mock 什么。

**怎么使用**：
- 测试必须断言**可观察行为**（输入 → 输出 / 副作用），而非内部实现细节（私有字段、调用次数、内部执行路径）。
- **理想模型必须转化为行为测试**，用测试定义"正确是什么"。这是 §1.6 复杂行为建模原则的可执行形式。
- **mock 仅用于外部依赖**（第三方 API、数据库、网络、时钟、随机数、文件系统），不用于内部模块边界。把被测对象的内部协作者都替换成 mock，等于在测试实现细节而非行为——过度 mock 内部依赖视为违规。
- 时序不变量通过**行为断言**验证（如事件时间戳的先后、间隔），而不是断言内部调用了某方法 N 次。
- 可观测性不足导致无法写行为测试时，先重构增加可观测性（暴露事件、时间戳、状态查询接口），再写测试。
- 这些测试能否进入 CI 门禁、CI 接受 / 拒绝哪些断言，由 §9.5 CI/CD 断言原则统一规定；本节只规定**怎么写**行为测试。

**小例子**：
> ❌ `assert mock.call_count == 3`——断言调用次数，是内部实现细节，重构即脆裂。
> ✅ `assert result.status == "paid"`——断言可观察的输出，是用户可感知的行为。

### 6.4 时序测试铁律 [error]

**使用条件**：测试异步 / 并发模块的时序行为时。

**怎么使用**：
- **注入可控延迟**：禁止仅用零延迟 mock 验证时序行为。必须通过 fake clock、可控 await、显式 sleep 等手段注入可调延迟，覆盖"快路径"和"慢路径"两种情形。
- **断言时序不变量**（顺序、间隔、超时边界），而非内部调用次数。不变量例子："通知推送时间戳 ≤ LLM 收到时间戳"、"重试间隔 ≥ 退避下限"。
- **关键时序不变量测试必须标记 `@pytest.mark.timing`**（或等价的框架标记），以便 CI 单独调度与门禁（见 §9.4）。
- 时序测试禁止依赖真实墙钟的绝对值；断言必须基于受控的相对时间或注入的 fake clock。

**小例子**：
> 测"重试间隔 ≥ 退避下限"，不能用零延迟 mock（间隔永远是 0，证明不了任何东西）。要用 fake clock 在重试之间注入 100ms 延迟，断言间隔 ≥ 退避下限。

## 七、Git 工作流

### 7.1 Conventional Commits 1.0.0

格式：`<type>[scope]: description`

| type | 用途 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档变更 |
| style | 代码格式（不影响功能） |
| refactor | 重构（不新增功能、不修复Bug） |
| perf | 性能优化 |
| test | 测试相关 |
| build | 构建系统/依赖变更 |
| ci | CI配置变更 |
| chore | 杂项（不修改src或tests） |
| revert | 回退之前的commit |

- 破坏性变更用 `!` 标记或 body 中 `BREAKING CHANGE:` 前缀
- scope 可选，表示影响范围（如 `auth`, `api`）
- description 简明扼要，小写，不加句号

### 7.2 分支策略

- `main`：生产分支，只接受合并，不直接提交
- `develop`：开发分支
- `feature/*`：功能分支
- `fix/*`：修复分支
- PR 必须通过所有检查才能合并

## 八、反模式清单（明确禁止）

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 直接 mutate state | 导致状态不可预测，测试不稳定 | 使用不可变更新（如扩展运算符、immer、不可变数据结构） |
| 硬编码密钥/URL | 安全风险，环境切换时易出错 | 使用环境变量或配置文件，绝不写在代码里 |
| 静默吞异常 | 隐藏错误，让问题无法追踪 | 捕获后要么处理（含日志），要么向上传播；空 catch 块必须注释说明理由 |
| 无测试提交 | 新增代码无测试覆盖，回归风险高 | 任何新增功能或逻辑变更必须附带对应测试，门禁 3 直接校验 |
| 破坏性静默更改 | 不兼容的 API/接口变更未标注，导致依赖方崩溃 | 破坏性变更必须用 `BREAKING CHANGE:` 前缀标注，并在 PR 描述中说明迁移方法 |
| 手写标准库功能 | 重新实现语言/框架已提供的方法，增加维护成本 | 先查阅标准库文档，优先使用内置方法（如 `Array.find` 而非手写循环查找） |
| 重复造轮子 | 已有内部库或工具函数可用却自己实现 | 写代码前使用 enhanced_search 搜索现有实现，复用优先 |
| 无注解的 any / @ts-ignore | 绕过类型系统，隐藏真实问题 | 若确实无法正确标注（如第三方类型缺陷），必须加 `// HACK: <原因> — <触发条件或计划修复时间>`，审查 Agent 检查此项 |
| 隐性技术债务（无标记） | 故意留下的简化实现，但未标记，延期变永久 | 使用 `DEBT:` 标记，包含：(1) 妥协的具体内容，(2) 触发改进的条件（如"用户量 >10k 时重构"），(3) 时间上限（ceiling），缺一不可 |

### 8.0 软约束（审查 Agent 建议修改，不强制打回）

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 过早抽象（三振出局） | 只有一个使用场景就提前抽象，可能引入不必要的复杂度 | 允许立即抽象，但审查 Agent 会检查：抽象后是否让代码更清晰？是否引入难以理解的间接层？若否，建议回退 |
| 方案倒退（Bug 修复退回旧方案） | 修复 Bug 时回退到旧实现，可能只是逃避问题 | 允许使用旧方案，但必须在该次变更的说明中解释：为何旧方案在此场景下是正确的？且所有 AC 测试必须通过 |
| 调试日志残留 | console.log/print 等调试输出遗留在生产代码中 | AI 可在开发中使用，但任务结束前必须清理。若确需保留，必须添加注释说明目的（如 `// 监控：支付耗时统计，暂留`），否则门禁检查不通过 |

### 8.1 债务标记约定

借鉴 ponytail-debt 项目，采用 `DEBT:` 注释标记实现技术债务显性化。

**标记格式**：
```
// DEBT: <简化内容>。ceiling: <当前限制>。upgrade: <升级触发条件>。
# DEBT: <简化内容>。ceiling: <当前限制>。upgrade: <升级触发条件>。
```

**规则要求**：
- 所有故意留下的技术捷径必须使用 `DEBT:` 标记
- 每个标记必须包含 `ceiling`（当前简化方案的限制边界）
- 每个标记必须包含 `upgrade`（何时需要重新审视/升级的触发条件）
- 没有 `upgrade` 触发条件的标记视为高风险项（悄悄腐烂）

**示例**：
```python
# DEBT: 使用内存缓存，仅支持单实例。ceiling: 单进程内存上限。upgrade: 当部署多实例或缓存超过 1GB 时替换为 Redis。
```

```typescript
// DEBT: 密码校验仅做长度检查。ceiling: 无复杂度/泄露检测。upgrade: 当用户量超过 1000 或安全审计要求时接入 HaveIBeenPwned。
```

### 8.2 延期不永久原则

- **显性化**：技术债务必须在代码中可见标记，禁止隐性积累
- **有触发**：每个债务项必须有明确的 revisit 触发条件（用户量、数据量、功能变化等）
- **可追踪**：定期扫描 `DEBT:` 标记，生成债务清单
- **定期审查**：在代码审查时检查现有债务项是否已触发升级条件

### 8.3 解决方案阶梯

写新代码前，按以下顺序检查，选择最简可行方案：

1. **YAGNI**：这个功能/抽象真的需要存在吗？
2. **标准库**：标准库是否已经实现？
3. **原生平台特性**：是否有平台原生能力可用？
4. **已安装依赖**：已有依赖是否能解决？
5. **最少代码**：以上都不行，才写最少的代码

- 每跳一级需在代码审查时说明理由

## 九、质量门禁

### 9.1 Pre-commit Hook

提交前自动检查：
- Lint（ESLint / Ruff / mypy）
- 格式化（Prettier / Black）
- 类型检查（TypeScript / mypy）
- 快速测试（smoke test）

### 9.2 CI/CD 流水线

```
Lint → Type Check → Unit Test → Integration Test → Build → Deploy
```

- 任何阶段失败阻塞合并

### 9.3 量化阈值（唯一真相源）

以下阈值为全项目统一标准，其他规则文件引用此表，不再重复定义。

| 指标 | 阈值 | 说明 |
|------|------|------|
| 函数圈复杂度 | ≤ 10 | 过高则提取子函数 |
| 函数认知复杂度 | ≤ 15 | 过高则拆分 |
| 函数体行数 | ≤ 50 行 | 过长则拆分 |
| 文件行数 | 软参考 ~1000 行 | 出现 ≥2 个不相干职责时才拆分，行数非硬判据（见 §4.2） |
| 函数参数个数 | ≤ 4 | 超过则使用对象封装 |
| 嵌套层数 | ≤ 4 层 | 超过必须重构 |
| 代码重复率 | ≤ 5% | 重复代码必须提取 |
| 新增代码覆盖率 | ≥ 80% | 行覆盖 + 分支覆盖 |
| 新增代码重复率 | < 3% | 避免复制粘贴 |
| 新增 Bug 数 | 0 | SonarQube Bug 规则 |
| 新增漏洞数 | 0 | 安全漏洞 |
| 新增异味数 | < 5 | 代码异味（Code Smell） |
| 单元测试通过率 | 100% | 不允许失败用例 |
| Lint warning | 0 | `--max-warnings 0` |

### 9.4 关键不变量门禁

CI 必须包含独立的 **timing 测试 stage**，专门运行标记为 `@pytest.mark.timing`（见 §6.4）的用例。

- timing stage 失败**直接拦截合并**，与单元测试同级别，不可降级为 warning 或 `|| true` 容错。
- 该 stage 覆盖关键时序不变量（事件顺序、间隔、超时边界），破坏不变量的改动在 CI 阶段即被拦截。
- timing stage 必须与单元测试 stage **隔离运行**，避免因真实延迟导致的执行时间不确定影响主测试套件的稳定性。
- timing 测试应断言时序不变量（可观察行为），而非内部调度细节；判据见 §9.5。

### 9.5 CI/CD 断言原则 [error]

**使用条件**：决定哪些测试、哪些断言能进入 CI 门禁时。

**怎么使用**：

CI 的核心问题是"这次改动能安全上线吗"——即验证**系统对外承诺的契约是否成立**，而非验证代码内部怎么实现。

**契约 vs 噪声**——CI 价值的唯一判据：

| 契约类型 | 验证方式 | CI 价值 |
|----------|----------|---------|
| API 返回正确结果 | 输入 → 输出断言 | 高 |
| 事件顺序正确 | 时序不变量断言 | 高 |
| 用户可感知的副作用 | 可观察行为断言 | 高 |
| 内部函数调用了几次 | `mock.call_count` | **无（噪声）** |
| 私有方法是否被调用 | 实现细节断言 | **无（噪声）** |

**断言必须满足的双重性质**——任何 CI 门禁测试都必须同时成立：

```
✅ 任何不改变外部行为的内部重构，测试必须保持绿色
✅ 任何破坏外部行为的改动，测试必须变红
❌ 不改变外部行为的重构导致测试变红 → 测试写得差
```

**分轨管理**：
- **常规 CI 测试**：只断言行为，不碰内部实现。`mock` 仅用于外部依赖（第三方 API、数据库、网络、时钟、随机数、文件系统），不用于内部模块边界。
- **白盒诊断测试**：可以断言内部状态，但**不进入 CI 门禁**——放开发阶段手动运行辅助定位，修完 bug 后删除，或移入非门禁的诊断套件。

> 内部断言（`mock.call_count`、私有方法、内部状态）进入 CI 会导致：重构误报、维护成本倒挂、噪声掩盖真实问题、不回答"能上线吗"。禁止它们进入门禁。

**小例子**：
> ❌ CI 里跑 `assert scheduler.run.call_count == 3`——重构一次调度器就误报，但用户根本感知不到调度次数。
> ✅ CI 里跑 `assert notification_pushed_ts <= llm_request_ts`——验证用户可感知的事件顺序，重构不误报，行为破坏必变红。
