# 调研领域规则

---

## 一、核心原则

### 1.1 问题驱动，不是搜索驱动

- 调研始于明确的**问题列表**，不是"先搜搜看"
- 每次搜索必须对应一个具体的调研问题
- 搜索结果必须映射回问题列表，标注覆盖状态

### 1.2 信源分级，交叉验证

信源等级定义见 information_integrity_rules.md

- 关键结论至少2个独立来源支撑
- 无法判断等级时默认标为C级
- 调研结论中至少50%的支撑信源应为A级或B级

### 1.3 收集-写入循环

- 每收集5-10条信息就写入中间笔记
- 避免无限收集（"再搜一条"陷阱）
- 每轮自查覆盖度：已回答/未回答/部分回答

### 1.4 结论可追溯

- 每个关键发现必须附来源链接
- 矛盾信息不隐藏，明确标注并说明判断依据
- 最终报告中的结论可在原文中定位

## 二、调研工作流（五段式）

```
1. 方向校准 → 2. 信息收集 → 3. 闭环校验 → 4. 报告撰写 → 5. 自评审计
```

### 2.1 方向校准

**目标**：将调研目标拆解为可执行的问题列表。

1. 分析调研目标，识别核心信息需求
2. 拆解为3-7个子方向
3. 每个子方向细化为3-5个具体问题
4. 写入问题清单（`research_questions.md`）
5. 确定搜索策略和优先级

**问题设计原则**：
- 每个问题应当可以通过一次搜索回答
- 问题之间不重复
- 覆盖"是什么/怎么做/为什么/对比什么"四个维度

### 2.2 信息收集

**目标**：按问题列表逐一搜索，收集原始信息。

**搜索策略**（按优先级）：
1. **网络搜索**（web_search）→ 广覆盖，快速定位方向
2. **网页抓取**（fetch）→ 精准抓取官方文档/GitHub README
3. **记忆检索**（memory）→ 查找历史调研经验
4. **文档分析**（file_read）→ 读取已有文档/设计文档

**搜索技巧**：
- 英文关键词往往比中文结果质量更高
- 添加"official"、"specification"、"README"等限定词提升精度
- GitHub搜索时加`language:xxx`、`stars:>1000`过滤
- 遇到SEO污染时，直接fetch官方域名

### 2.3 闭环校验

**目标**：确保所有问题已回答，无遗漏。

1. 对照问题清单逐项检查覆盖状态
2. 标注：✅已回答 / ⚠️部分回答 / ❌未回答
3. 对未回答的问题，分析原因并决定：
   - 可以补充搜索 → 回到2.2继续收集
   - 无法获取 → 在报告中标注"信息不足"
4. 对矛盾信息进行判断，说明依据

### 2.4 报告撰写

使用 `config/templates/research_report_template.md` 输出结构化调研报告，章节结构以模板为准，不在本规则重复。

### 2.5 自评审计

**目标**：自检报告质量。

| 检查项 | 通过标准 |
|--------|----------|
| 问题覆盖度 | 所有问题已回答或标注"信息不足" |
| 信源分布 | A+B级占比 ≥ 50% |
| 交叉验证 | 关键发现至少2个来源 |
| 矛盾处理 | 矛盾信息已标注并说明判断依据 |
| 格式规范 | 报告对照 research_report_template.md，章节齐全 |

## 三、调研类型与策略

### 3.1 技术调研

**关注点**：技术方案可行性、性能、兼容性、社区活跃度。

**搜索优先级**：官方文档 > GitHub仓库 > 技术博客 > 社区问答

**必查项**：
- 版本号和发布日期（是否过时）
- Star数/Fork数（社区活跃度）
- 已知Issue（潜在风险）
- 与竞品的对比基准

### 3.2 资源调研

**关注点**：已有资源清单、复用可行性、缺口分析。

**搜索优先级**：项目文档 > 外部资源

**必查项**：
- 项目文档中已有同类资源（避免重复建设）
- 资源质量和维护状态
- 复用/适配成本
- 资源缺口清单

### 3.3 方案调研

**关注点**：多个方案的结构化对比、优劣势分析、推荐理由。

**搜索优先级**：行业报告 > 官方对比文档 > 实践案例 > 社区评价

**必查项**：
- 至少3个可选方案
- 每个方案的完整优劣势分析
- 基于场景的推荐（不同场景可能适合不同方案）

### 3.4 市场调研

**关注点**：市场规模、竞争格局、用户需求、趋势分析。

**搜索优先级**：行业报告 > 官方数据 > 新闻报道 > 社区讨论

**必查项**：
- 数据时效性（标注数据年份）
- 数据来源可信度
- 是否有矛盾数据（不同来源差异大时重点标注）

## 四、调研报告质量标准

### 4.1 必须满足

- [ ] 所有调研问题已覆盖（100%覆盖度）
- [ ] 每个关键发现附来源链接和信源等级
- [ ] 关键发现至少2个独立来源支撑
- [ ] 矛盾信息已标注并说明处理方式
- [ ] 结论可在原文中追溯
- [ ] 报告对照 research_report_template.md，必填章节齐全

### 4.2 建议满足

- [ ] 术语表（专业术语解释）
- [ ] 风险提示（调研结论的不确定性）

## 五、反模式清单

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| 无问题列表就搜索 | "先搜搜看" | 先拆解问题，再按问题搜索 |
| 只搜不写 | 收集几十条信息不整理 | 每收集5-10条就写入笔记 |
| 无限搜索 | "再搜一条可能更好" | 设定搜索次数上限，到量整理 |
| 隐藏矛盾 | 只展示支持结论的来源 | 矛盾信息明确标注，说明判断依据 |
| 信源无标注 | 来源不明确 | 每条信息标注信源等级和链接 |
| 单一来源 | 关键结论只有一个来源 | 关键发现至少2个独立来源交叉验证 |
| 结论无追溯 | 报告中的数字/事实无法定位到原文 | 每个结论附原文链接或位置 |
| 忽略时效性 | 引用3年前的数据说明"当前状况" | 标注数据年份，评估时效性 |
| SEO内容当权威 | 把CSDN/知乎列为A级来源 | 官方文档/GitHub才是A级 |

## 六、搜索关键词优化

### 6.1 关键词构造原则

- **具体 > 宽泛**：`Cursor .cursorrules MDC format` 优于 `AI coding rules`
- **英文 > 中文**（技术领域）：英文结果通常质量更高
- **加限定词**：`official`、`specification`、`vs`、`benchmark`、`README`

### 6.2 搜索词模板

| 调研类型 | 搜索词模板 |
|----------|-----------|
| 工具调研 | `[工具名] official documentation features` |
| 规范调研 | `[规范名] specification standard RFC` |
| 对比调研 | `[A] vs [B] comparison benchmark` |
| 最佳实践 | `[领域] best practices guide [年份]` |
| 开源项目 | `site:github.com [关键词] stars:>1000` |

## 七、中间文档管理

调研过程中产出的中间文档应有序管理：

```
docs/working/
├── research_questions.md      # 调研问题清单
├── research_todos.md           # 任务执行checklist
└── reference_materials/
    └── [domain]/               # 按领域分目录
        ├── README.md           # 参考材料目录索引
        └── [topic].md          # 按主题整理的参考资料
```

**命名规范**：
- 中间文档放 `docs/working/`
- 参考材料放 `docs/working/reference_materials/[domain]/`
- 最终报告放 `docs/`（如 `docs/编程领域规则标杆调研_research_report.md`）

## 八、调研方法论：来源降级

当外部调研工具不可用（如网络搜索后端故障、目标网站不可达）时，按以下策略降级获取来源。

### 8.1 两层降级

| 层级 | 条件 | 做法 | 标注方式 |
|------|------|------|----------|
| 第一层：外部调研 | 搜索/抓取可用 | 正常执行网络调研，记录来源URL | `[来源: URL]` |
| 第二层：记忆 | 外部工具不可用 | 记忆检索历史调研经验 | `[来源: 记忆]` |
| 无材料终止 | 记忆也无相关经验 | 不编写该条内容 | 无材料处理见 information_integrity_rules.md |

### 8.2 降级执行

明确说明实际使用了哪一层降级策略（第一层 / 第二层 / 无材料终止）。
