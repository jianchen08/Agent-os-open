# 上下文读取与文档规范

> 工作时请遵循以下原则：先读文档理解上下文，再动手执行；创建文档按规范，保持文档与实际一致。

---

## 一、执行前：先读文档再动手

**信息获取顺序**：确定模块 → 项目级文档 → 模块文档 → 组件文档 → 源码（仅在需要时）

**索引使用**：每层文档的"结构"章节是索引入口，通过索引定位，不要盲目搜索。

**上下文范围**：
- 单模块 → 该模块文档 + 项目级（如需依赖）
- 多模块 → 所有相关模块文档 + 项目级
- 全局架构 → 项目级 + 所有受影响模块

---

## 二、项目文档在哪里

项目文档统一存放在 `.project/` 目录，**全部平铺不分子文件夹**：

| 文件类型 | 命名 | 性质 |
|----------|------|------|
| 宪法文档 | `{type}_charter.md` | 冻结，项目最高准则 |
| 项目规则 | `{type}_rules.md` | 跟着项目走的特定规则 |
| 其他文档 | 自由命名 | 架构、大纲、API设计等 |

**关键区分**：
- `.project/` = 项目级文档，执行任务前检查是否有相关文档
- `docs/working/` = 中间过程文档，用于存放调研笔记、草稿等临时产出

---

## 三、创建/修改文档：按规范来

| 层级 | 位置 | 内容 |
|------|------|------|
| 项目级 | `.project/` | 架构、API、配置、宪法 |
| 模块级 | `{模块}/{模块}.md` | 需求、逻辑、结构 |
| 组件级 | `{模块}/{组件}/{组件}.md` | 需求、逻辑、结构 |

**四阶段**：需求 → 逻辑 → 结构 → 实现（指向下层）

**规则**：
1. 修改接口/配置时必须同步更新文档
2. 新增文件时添加"结构 > 文件清单"条目
3. 新增组件/模块时创建对应文档
4. 禁止在文档中写具体实现代码
5. 文档跟随代码（放在模块代码同层）
