# 状态机测试设计流程规范

---

## 1. 状态机测试设计流程

### 1.1 总体流程

```
识别状态 → 识别转换 → 设计测试 → 验证覆盖 → 补充遗漏
```

### 1.2 步骤一：识别状态

**目标**：找出组件/模块中所有用户可感知的状态。

**识别方法**：

1. 从结构化需求的 state_machine 中获取已定义的状态列表
2. 如需求未提供状态机，则从代码中识别：
   - 找出所有异步操作 → 每个异步操作至少有 loading/success/error 三态
   - 找出所有条件渲染 → 每个条件分支对应一个状态
   - 找出所有业务规则（如"连续失败 N 次锁定"）→ 锁定是一个独立状态
3. 确保覆盖：初始态、进行中态、成功态、失败态、特殊业务态

### 1.3 步骤二：识别转换

**目标**：找出状态之间的所有转换关系。

**识别方法**：

1. 对每对相邻状态，确认：什么事件触发转换？转换后的可见效果是什么？
2. 检查是否有守卫条件（guard）：某些转换只在特定条件下发生
3. 检查是否有副作用（side effect）：转换时是否触发其他行为

### 1.4 步骤三：设计测试

**目标**：为每个状态和转换设计测试用例。

**设计原则**：

- 每个状态至少一个测试：验证进入该状态后的可见效果
- 每个转换至少一个测试：验证触发事件后状态正确变化
- 每个守卫条件至少两个测试：正向（条件满足，转换成功）和反向（条件不满足，转换被阻止）

**测试必须验证的要素**：

| 要素 | 说明 |
| ---- | ---- |
| 状态值 | 转换后组件/模块的状态是否正确 |
| 可见效果 | 对应的 UI 表现是否正确（如按钮 disabled、文案变化、元素显隐） |
| 副作用 | 转换时是否触发了预期的附加行为 |

### 1.5 步骤四：验证覆盖

**目标**：确认测试覆盖了所有状态和转换。

**覆盖率定义**：

- **state_coverage** = 已测试的状态数 / 已识别的总状态数
- **transition_coverage** = 已测试的转换数 / 已识别的总转换数

**通过标准**：

- state_coverage = 100%
- transition_coverage = 100%
- 所有 must 级别的验收标准都有对应测试

### 1.6 步骤五：补充遗漏

对照步骤一和步骤二的结果，检查是否有遗漏的状态或转换未测试。如有，补充测试用例。

## 2. 常见状态机模式（参考）

以下模式用于辅助识别，不是固定模板。具体状态取决于业务需求：

### 2.1 数据获取模式

涉及组件：列表页、详情页、任何从服务端获取数据的场景。

通常涉及的状态：idle → loading → (success / empty / error)

关键检查点：loading 时有无反馈？空数据时有无占位？错误时有无重试？

### 2.2 表单提交模式

涉及组件：登录、注册、设置、任何需要提交数据的场景。

通常涉及的状态：idle → validating → submitting → (success / error)

关键检查点：提交中是否防重复？校验失败是否有定位？提交失败是否可重试？

### 2.3 条件锁定模式

涉及组件：密码重试、限流、任何有次数/时间限制的场景。

通常涉及的状态：idle → counting → locked → (timeout → idle)

关键检查点：计数是否准确？锁定时是否阻止所有操作？解锁后是否恢复正常？

### 2.4 确认操作模式

涉及组件：删除、支付、任何不可逆操作。

通常涉及的状态：idle → confirming → (confirm → executing → success / error / cancel → idle)

关键检查点：确认弹窗内容是否具体？取消是否回到原状态？执行中是否防重复？

## 3. 状态机测试质量要求

> 状态识别见 requirement_structuring_rules.md，本文件只负责基于状态机的测试设计。
> 通用测试质量标准（独立性、命名、覆盖标准等）见 testing_rules.md。

### 3.1 多维度断言 [warning]

状态机测试中每个测试用例必须验证：
- **状态值**：转换后组件/模块的状态是否正确
- **可见效果**：对应的 UI 表现是否正确（如按钮 disabled、文案变化、元素显隐）
- **状态变化**：不仅验证返回值，还要验证状态变化
- 避免单一断言测试：一个测试应验证完整的状态转换行为

### 3.2 回归测试根因验证 [error]

回归测试必须验证**根本原因**被修复，而不仅仅是**症状**消失：
- 测试验证的是根因条件，不是症状消失
- 测试有明确的断言消息，说明断言的是什么
- 如果根因是状态转换问题，测试必须验证状态转换的正确性，不仅仅是最终值
