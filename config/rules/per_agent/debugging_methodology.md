# Bug 调试方法论

## 总原则

**先确认"谁在执行"，再优化"怎么执行"。**

绝大多数深层 bug 不在报错代码本身，而在对象生命周期或编排层。直接在报错位置反复修改是最大的时间浪费。

### 强制流程（跳步视为流程违规）

```
规则 0（写失败测试）
  → 规则 8（收集数据）
  → 规则 7 / 9 / 12（分析）
  → 规则 11（写回归测试并验证根因）
  → 修改代码
```

本方法论遵循《编程领域规则》中的行为测试规范（§6.3 行为测试原则、§6.4 时序测试铁律），但提供具体的诊断手段。

---

## 规则 0：用测试捕获理想行为（前置硬门槛）

**使用条件**：发现 bug 的第一时间——任何分析代码、读日志、改逻辑之前。

**怎么使用**：
- **第一动作是写出能暴露 bug 的行为测试（红灯）**，而不是分析代码。该测试即理想模型的可执行形式，精确描述"在给定输入下，正确的可观察行为应该是什么"。
- 这条规则优先于规则 1～12 的所有分析动作。先有失败测试，再谈诊断。
- 测试必须断言**可观察行为**（输入 → 输出 / 副作用），而不是内部实现细节，遵循《编程领域规则》§6.3。
- 如果无法写出测试（例如缺乏可观测性），**不要绕过这一步**——先重构增加可观测性（暴露事件、时间戳、状态查询接口），再写测试。
- 写出红灯测试后，再进入规则 8 收集数据、规则 7/9/12 分析根因，最后用规则 11 的回归测试验证修复。

**小例子**：
> 通知推送时序错了，先不读代码。直接写一个测试断言"通知推送时间戳 ≤ LLM 收到请求时间戳"，跑一次，红灯——这就是 bug 的可执行形式。

```python
def test_notification_pushed_before_llm_call():
    """理想模型：通知推送时间戳必须早于 LLM 收到请求的时间戳"""
    pipeline = build_pipeline()
    await pipeline.run("hello")
    assert pipeline.events["notification_pushed_ts"] <= pipeline.events["llm_request_ts"]
    # 当前实现下此测试失败（红灯）→ 这就是 bug 的可执行形式
```

## 规则 1：自顶向下，不要自底向上

**使用条件**：拿到报错的第一时刻——尤其是有明确的异常类型/报错信息时。

**怎么使用**：
- 先定位**执行实例**（是哪个对象在跑），再定位代码行。报错位置常常是受害者，不是凶手。
- 在报错的最顶层入口加标记（print/logger），确认是哪个对象实例在执行。
- 在所有可能的分支/实现都加标记，而不是只在你"认为"正确的那个加。
- 标记内容至少包含：`type(self).__name__` 和关键参数。

**小例子**：
> 家里灯不亮，直接拆灯泡检查（自底向上）——但可能根本是电闸跳了。正确做法是先确认"是这一盏灯不亮，还是整层楼不亮"（定位范围），再决定查灯泡还是查电闸。

```python
# 手段之一：在所有适配器的 _do_completion 都加标记
class LiteLLMAdapter:
    async def _do_completion(self, **kwargs):
        print(f"[DIAG] adapter={type(self).__name__} model={kwargs.get('model')}")
        ...

class AdaptiveRouterAdapter:
    async def _do_completion(self, **kwargs):
        print(f"[DIAG] adapter={type(self).__name__} model={kwargs.get('model')}")
        ...
```

## 规则 2：隔离测试通过 ≠ 系统正确

**使用条件**：隔离测试（单元测试）通过了，但完整系统仍然失败时。

**怎么使用**：
- 隔离层验证"逻辑正确性"，系统层验证"编排正确性"，二者不等价。隔离测试通过绝不等于 bug 已修复。
- 隔离测试通过后如果 bug 仍在，**立刻停止在隔离层继续修改**，转向系统集成层。
- 经验触发线：隔离测试通过但系统仍失败时，最多再做 **1 轮**隔离测试确认，然后必须切全链路追踪——再多 1 轮不会增加"这段代码是否被调用"的新信息。

**小例子**：
> 汽车零件单独检测都合格，但组装后仍可能发动不了——零件正确不等于装配正确。检测单点合格后，仍须整车点火验证。

## 规则 3：追踪对象生命周期，而不是代码路径

**使用条件**：功能 A 在隔离测试中正常，但在系统中失败——典型的"代码对、运行时错"。

**怎么使用**：
- 追踪**实例的来源与重建路径**：它被构造了几次、由谁构造、运行时用的实例和你修改的是否是同一个。
- 在对象的**构造函数**加调用栈追踪，确认创建次数和创建者。
- 检查所有可能的工厂 / fork / clone / rebuild 路径。

**小例子**：
> 你以为在改张三的档案，但系统里其实有两个"张三"。改了半天没用，因为运行时用的是另一个张三。正确做法：先确认你手里这个张三的 id，再看它是从哪个入口被创建的。

```
本次 bug 的本质：
build_plugin_registry → LLMCore(adapter=AdaptiveRouterAdapter)  ✓
    ↓ registry.fork()
PipelineEngine.__init__ → LLMCore(config=...) → LiteLLMAdapter  ✗  ← adapter 丢失！
    ↓ _apply_agent_model_override
engine.run → 把 LiteLLMAdapter 当作 existing_adapter 传递      ✗

bug 不在 LLM 调用层，而在插件注册表的 fork 机制。
```

```python
# 手段之一：在构造函数追踪对象创建（limit 按需调整）
class LLMCore:
    def __init__(self, **kwargs):
        import traceback
        traceback.print_stack(limit=4)
        print(f"[DIAG] LLMCore created with adapter={type(kwargs.get('adapter')).__name__}")
```

## 规则 4：诊断要加在分叉点，不是执行点

**使用条件**：需要加日志诊断时——尤其在有多态、策略模式、插件注册表的系统里。

**怎么使用**：
- 日志加在**分叉点**（多个实现可能被执行的地方），而不是执行点（具体实现内部）。
- 如果分叉点选错了实现，执行点内的所有日志都是在错误的代码上打的，完全没有参考价值。
- 正确做法：在分叉点和创建点各加 1 行标记，胜过在执行点内部加 20 行详细日志。

**小例子**：
> 十字路口有四条岔路，你不知道车开去了哪条。正确做法是在路口（分叉点）装一个摄像头，而不是在某一条岔路上装 20 个。

## 规则 5：警惕 confirmation bias — 主动证伪，不要只验证

**使用条件**：每次修改后 bug 仍在时——防止陷入"越改越觉得快好了"的死循环。

**怎么使用**：
- 优先找**推翻**假设的证据，而不是找支持它的证据。主动证伪优于单向验证。
- 每当修改后 bug 仍在，问自己："如果我的假设是错的，最可能的替代解释是什么？"
- 经验触发线：同一方向尝试 **3 次**无效后，**强制切换假设**——在偏见固化前打断。
- 用 print 在错误对象上打标记，而不是在"应该是正确"的对象上打标记。

**小例子**：
> 医生只找支持"感冒"诊断的症状，会漏诊肺炎；好医生会专门找能推翻"感冒"的证据——发烧到几度？持续几天了？有没有胸痛？一旦发现矛盾症状，立即重新评估。

## 规则 6：插件/工厂系统的常见陷阱清单

**使用条件**：bug 表现为"功能在简单场景正常，但在完整流程失败"时。

**怎么使用**：按以下清单逐项检查：

| 检查项 | 常见问题 |
|--------|---------|
| **fork/clone** | 是否只复制了 config 而丢失了 adapter/router 等运行时依赖？ |
| **工厂重建** | `type(obj)(config=...)` 是否漏传了构造函数参数？ |
| **单例 vs 多实例** | 以为是同一个对象，实际被重建了？ |
| **依赖注入** | 依赖是通过构造函数注入还是运行时获取？注入的是引用还是值？ |
| **缓存/全局状态** | `__pycache__`、模块级单例是否过期？ |
| **注册表同步** | 替换插件后，core_plugins 和 plugins 两个映射是否都更新了？ |

**小例子**：
> 你给每个房间都配了钥匙，但管家换班时只交接了 config（房间号），没交接钥匙本身——新管家拿着房间号却开不了门。

## 规则 7：根因分析 — 逐层追问（5 Whys）

**使用条件**：每次修复后 bug 仍在时；或发现自己开始在同一方向打转时。

**怎么使用**：
- 从表层症状逐层追问"为什么"，追到**可操作根因**为止——即改了它，bug 不会以任何形式复现。
- 典型因果链深度 4-5 层，每层各追一类信息：

| 追问层 | 从 ___ 追到 ___ | 回答的问题 |
|--------|----------------|-----------|
| Y1 | 症状 → 直接原因 | 表面错误是什么导致的？ |
| Y2 | 直接原因 → 触发条件 | 这个直接原因在什么条件下发生？ |
| Y3 | 触发条件 → 运行时机制 | 系统运行时哪个机制制造了这个条件？ |
| Y4 | 机制 → 设计选择 | 这个机制为什么这样设计 / 漏了什么？ |
| Y5 | 设计 → 根本原因 | 根本的设计 / 流程缺陷是什么？ |

- **终止条件**（比"问够 5 次"更重要）：到达可操作根因即停，不必凑满 5 次；3 层追到根因就停。超过 5 层仍无根因 → 转规则 12（多维度并行排查）。
- 每一层的答案必须是**事实**（日志/堆栈/数据），不是猜测——偏差会跨层累积。

**小例子**：
> 迟到 →（Y1）起晚了 →（Y2）闹钟没响 →（Y3）手机昨晚被设成静音 →（Y4）开会时手动静音忘了恢复。到 Y4 就是**可操作根因**：建立"会议结束自动恢复铃声"的流程，bug 根除。不必硬凑 Y5。

**项目示例**：
```
1. 为什么报 AuthenticationError？→ API key 验证失败
2. 为什么 API key 验证失败？→ LiteLLMAdapter 使用了默认 key
3. 为什么使用了 LiteLLMAdapter？→ registry.fork() 重建时没传 adapter 参数
4. 为什么没传 adapter 参数？→ fork 只复制了 config，不知道有 adapter 属性
5. 为什么只复制 config？→ fork 方法设计时没考虑运行时注入的依赖

根因：fork 机制不支持运行时依赖的传递 → 修复 fork，而非修复 API key。
```

## 规则 8：先收集数据，再分析

**使用条件**：形成任何假设之前——尤其是想"我觉得问题出在……"的时候。

**怎么使用**：
- 假设必须基于**事实数据**，而不是直觉。数据收集阶段不改代码（只加诊断）。
- 经验触发线：至少收集 **2 轮**数据后再形成假设——单轮数据可能有偶然性，2 轮交叉验证是建立可信假设的最小样本。

**数据收集清单**：

| 数据类型 | 收集方式 | 目的 |
|---------|---------|------|
| 完整错误堆栈 | 捕获 traceback | 定位出错位置 |
| 对象类型和ID | `print(type(x).__name__, id(x))` | 确认是哪个实例 |
| 输入参数快照 | 打印关键参数值 | 排除参数错误 |
| 执行路径标记 | 在分叉点加 print | 确认走了哪条路径 |
| 对象创建栈 | `traceback.print_stack(limit=4)` | 追踪对象生命周期 |

**小例子**：
> 医生先做检查（验血、拍片）再下诊断，不会瞄一眼就开药。没有检查结果的诊断是猜测，不是诊断。

## 规则 9：证据驱动的假设管理

**使用条件**：开始形成多个假设，需要决定优先验证哪个时。

**怎么使用**：
- 假设收敛、证据驱动。每个假设同时跟踪正面和反面证据；**排除假设比确认假设更有价值**——被排除的假设不用再试。
- 经验触发线：同时维护的假设**不超过 3 个**——这是有效跟踪正反证据的上限。如果 3 个假设都被排除，不要硬造第 4 个，而是**回到数据收集阶段**（规则 8）——大概率遗漏了关键信息。

**假设模板**：

```
假设 H1：<描述>
  正面证据：<支持此假设的事实>
  反面证据：<与此假设矛盾的事实>
  验证方式：<如何证明/证伪>
  状态：待验证 / 已排除 / 已确认
```

**小例子**：
> 侦探不会同时追 10 个嫌疑人，而是聚焦 2-3 个最可疑的，分别列出"支持是他"和"推翻是他"的证据。排除一个嫌疑人比锁定一个更有价值——少走一条死路。

## 规则 10：架构级熔断器

**使用条件**：修复尝试反复失败、开始怀疑问题不在代码层时。

**怎么使用**：
- 当修复尝试反复失败时，问题大概率不在代码层而在架构层。失败累积到阈值，**停止改代码，升级到架构重新评估**。
- 经验熔断线（满足任一即触发）：同一代码位置修改约 **3 次**仍未通过 / 同一假设方向尝试约 **3 次**无效 / 总修复轮次累计约 **6 次**仍未解决。
- 熔断后：停止所有代码修改 → 重新评估架构（模块关系、数据流、对象生命周期）→ 标记"bug 发生位置"和"修复尝试位置"→ 若两者不在同一层，问题在架构层而非代码层。
- 画图是辅助手段，非强制产物，结论也可以通过测试/数据/日志得出。

**架构级问题的特征**：bug 在简单场景不出现只在完整流程出现 / 隔离测试通过但集成测试失败 / 同一段代码在不同上下文中行为不同 / 对象/模块交互与文档或假设不一致。

**小例子**：
> 修水龙头连换 3 个垫圈还漏水，该停下想"是不是水管接口裂了 / 水压太高"，而不是继续买第 4 个垫圈。问题已经不在垫圈这一层了。

## 规则 11：回归测试的正确写法

**使用条件**：定位到根因、准备写回归测试时——验证修复是否真的解决了根本原因。

**怎么使用**：
- 回归测试**必须是行为测试**——验证用户 / 下游可感知的结果（输入 → 输出 / 副作用），而非内部实现细节。同时必须验证**根本原因**被修复，而不是**症状**消失。
- 测试验证的是**根因条件**，不是症状消失；测试必须有明确的**断言消息 / docstring**，说明断言的是什么不变量。遵循《编程领域规则》§6.3。
- **若根因是对象生命周期问题**，优先通过可观察行为捕获；仅在行为层无法稳定捕获时，才用内部断言作为补充诊断，且必须：用 `@pytest.mark.whitebox` 显式标注 + 注明"此断言依赖内部结构 X，重构时需同步更新"+ 仍需附带至少一条行为断言作为主判据。
- **CI 入门资格**：只有行为测试能安全进入 CI 门禁。`whitebox` 标记的内部诊断断言**不得进入 CI 门禁**，放开发阶段手动运行，修完 bug 后删除或移入非门禁诊断套件。判据见《编程领域规则》§9.5。

**小例子**：
> ❌ 错误：`assert mock.call_count == 3`——断言调用次数是断言实现细节，重构即脆裂。
> ✅ 正确：`assert events["notification_pushed_ts"] <= events["llm_request_ts"]`——断言可观察的时间戳先后，是用户可感知的契约。

**反例与正例**：

```python
# ❌ 错误：只验证症状，不是行为测试
def test_no_auth_error():
    result = llm.call("hello")
    assert result is not None  # 通过了，但可能只是换了另一种方式失败

# ❌ 错误：断言实现细节，不是行为
def test_fork_called_three_times():
    assert mock.call_count == 3

# ✅ 正确：验证可观察行为 + 根因
def test_notification_pushed_before_llm_call():
    """根因：通知必须在 LLM 调用前推送。
    可观察行为：通知推送时间戳 ≤ LLM 收到请求时间戳。"""
    pipeline = build_pipeline()
    await pipeline.run("hello")
    assert pipeline.events["notification_pushed_ts"] <= pipeline.events["llm_request_ts"]
```

## 规则 12：鱼骨图分析法

**使用条件**：逐层追问（规则 7）超过 5 层仍找不到明确根因时——线性因果链失效。

**怎么使用**：
- 切换为**多维度并行排查**——不再追问"为什么"，而是从多个独立维度同时扫描。
- 按以下分类逐项排查（**常用维度示例，非穷举，可按项目实际增减**）：

```
                    Bug 症状
                       |
        ┌──────┬───────┼───────┬──────┐
        |      |       |       |      |
     代码层   数据层   环境层   编排层   配置层
```

| 分类 | 检查内容 |
|------|---------|
| **代码层** | 逻辑错误、边界条件、类型不匹配、异常未捕获 |
| **数据层** | 数据格式变化、缺失字段、编码问题、时序竞争 |
| **环境层** | 依赖版本、环境变量、文件权限、网络连通性 |
| **编排层** | 对象生命周期、插件注册、依赖注入、初始化顺序 |
| **配置层** | 配置文件格式、路径解析、默认值覆盖、合并策略 |

**小例子**：
> 找钥匙，桌上从左到右翻一遍找不到（线性扫描，对应规则 7）。换个思路：同时回忆"上次在哪间房、穿哪件衣服、几点回家、谁也在场"（多维度并行，对应鱼骨图）——后者覆盖面更广。

## 规则 13：时序 bug 的测试设计

**使用条件**：bug 涉及异步、并发、事件顺序、超时等时序问题时。引用《编程领域规则》§6.4 时序测试铁律。

### 13.1 如何注入可控延迟

- **禁止仅用零延迟 mock** 验证时序。零延迟下"顺序正确"不能证明任何时序不变量。
- 注入手段优先级：
  1. **fake clock**：注入可控的时钟（如 `freezegun`、自实现的 `Clock` 协议），让 `now()` / `sleep()` 由测试驱动。
  2. **可控 await 点**：在被测协程的关键节点插入可被测试手动推进的 `asyncio.Event` / `asyncio.Future`。
  3. **显式 sleep**（最后手段）：仅在 fake clock 不可行时使用，且必须用较小且确定的值，断言相对关系而非绝对耗时。
- 测试必须覆盖**快路径**（无延迟）和**慢路径**（关键节点被人为放慢）两种情形。

### 13.2 如何设计时间戳记录

- 关键事件**必须暴露可观察的时间戳**（事件名 → 时间戳的映射），供行为断言读取。若被测对象没有暴露，先重构增加可观测性（规则 0）。
- 时间戳必须来自**同一个受控时钟**（fake clock 或注入的 clock），禁止混用真实墙钟。
- 记录位置应在**事件实际发生点**（发送 / 接收边界），而不是内部状态变更点。

### 13.3 时序不变量 → 行为断言的转换方法

把时序 bug 的根因转成可断言的不变量，遵循"先写失败测试"（规则 0）：

| 根因（理想模型） | 行为断言（可执行形式） |
|------|------|
| 通知必须在 LLM 调用前推送 | `assert events["notification_pushed_ts"] <= events["llm_request_ts"]` |
| 重试间隔不小于退避下限 | `assert (retry[i+1] - retry[i]) >= backoff_floor` |
| 超时后必须取消下游 | `assert cancel_emitted_ts - start_ts <= timeout_bound` |

关键时序不变量的测试**必须标记 `@pytest.mark.timing`**，以便 CI 单独调度（见《编程领域规则》§9.4）。时序不变量属于可观察行为，按本规则设计的 timing 测试可直接进入 CI 门禁，判据见《编程领域规则》§9.5。

### 13.4 分层测试策略

- **单元层**：用 fake clock 验证单个协程 / 状态机的时序不变量。
- **集成层**：用可控 await 点驱动多个组件交互，验证跨组件的事件顺序。
- **端到端层**：仅验证少量、稳定的端到端时序不变量；避免对真实墙钟的精确值做断言。

**小例子**：
> 测"通知必须先于 LLM 调用推送"，不能只 mock 掉 LLM（零延迟下谁先谁后没意义）。要用 fake clock 在两个动作之间注入可控延迟，分别测"LLM 秒回"和"LLM 卡 5 秒"两种情况，都断言通知时间戳更早。

## 关于"画图"的定位（可选思维工具，非强制步骤）

"画理想时序图""画架构图"等手段**不是强制步骤**，而是可选的中间思维工具：

- 它们可以帮助思考理想行为、辅助沟通，但不作为合规产物，不进入规则门禁。
- **规则强制要求的只有"先写行为测试"**（规则 0）——理想模型必须以可执行的失败测试形式存在，而非图示或自然语言描述。
- 规则 10（架构级熔断器）中"画出当前架构图"同样定位为辅助手段：真正的判断依据是"修复尝试位置与 bug 发生位置是否在同一层"，这一结论可以通过测试、数据、日志等可执行手段得出，画图只是帮助形成该结论的工具。

**最终核心**：理想模型通过行为测试表达，测试就是规范。画图只是思维过程，规则强制的只有"先写测试"。
