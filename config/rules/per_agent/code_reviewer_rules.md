# 第一部分：通用规则

## 0. 法定审查步骤（总纲§三(二)，优先级最高）

以下三个法定审查步骤必须在每次代码审查中执行，拥有一票否决权：

### 0.1 需求追溯审查

- **需求依据检查** \[error]: 每一行新增代码必须能在需求文档或AC清单中找到依据
- **无需求代码标记** \[error]: 无依据的代码标记为"无需求代码"，强制删除
- **防御性逻辑检查** \[error]: 是否存在无需求依据的兼容、扩展、防御性代码

### 0.2 架构边界四问

- **散点检查** \[error]: 同一业务概念是否在多个文件中重复出现？若是，驳回并要求统一抽象
- **分叉点检查** \[error]: 调用方是否因同一操作需要判断多种前置状态？若是，驳回并要求将分叉点封装在模块内部
- **信息泄漏检查** \[error]: 模块是否知道了另一个模块的实现细节？若是，驳回并强制隔离
- **变化方向检查** \[error]: 接口是否暴露了易变的实现细节？稳定业务动作是否与易变技术实现耦合？若是，驳回并要求分离

### 0.3 铁面裁决原则 

- **客观标准** \[error]: 审查结论必须基于规则和自动化工具的客观结果，禁止主观通融
- **禁止降级** \[error]: Must Fix 级别问题不可通过辩解降级为 Should Fix 或 Nit
- **禁止放水** \[error]: 不因"改动不大""看起来还行""为了兼容性"而放过问题

### 0.4 冗余与质量审查

- **翻译式注释** \[error]: 代码注释是否仅翻译代码本身（如 `// set name to x`），应删除
- **无效错误处理** \[error]: 是否存在空 catch 块、仅日志记录的错误处理
- **死代码** \[error]: 是否存在未使用的变量、函数、导入、分支
- **风格不一致** \[warning]: 代码风格是否与项目既有风格一致

## 1. Google Code Review八大维度

### 维度1：Design（设计）

- **架构合理性** \[error]: 代码设计是否良好、是否适合系统架构
- **模块划分** \[error]: 功能是否放在正确的模块/层级中
- **扩展性** \[warning]: 设计是否便于未来扩展
- **接口设计** \[warning]: 公共接口是否清晰、最小化

### 维度2：Functionality（功能）

- **行为正确性** \[error]: 代码行为是否符合需求预期
- **边界情况** \[error]: 是否处理了边界条件和异常情况
- **用户价值** \[warning]: 变更是否对用户有益
- **副作用** \[error]: 是否有意外的副作用

### 维度3：Complexity（复杂度）

- **可读性** [error]: 其他开发者能否快速理解代码
- **抽象层次** [warning]: 抽象是否恰当，不过高也不过低
- **过度工程** [warning]: 见「过度工程审查」

### 维度4：Tests（测试）

- **测试覆盖** \[error]: 是否有正确且设计良好的自动化测试
- **测试质量** \[error]: 测试是否真正验证了功能，而非假性测试
- **测试命名** \[warning]: 测试命名是否清晰描述测试场景
- **缺失测试** \[warning]: 是否缺少边界条件或异常场景的测试

### 维度5：Naming（命名）

> 命名规范见「命名规范」。审查时重点检查命名是否表达意图、是否一致。

### 维度6：Comments（注释）

> 注释规范见「编码规范」。审查时重点检查注释是否解释"为什么"而非"做什么"、是否过时。

### 维度7：Style（风格）

> 风格规范见「代码风格」。审查时重点检查风格一致性。

### 维度8：Documentation（文档）

> 文档规范见「编码规范」和「文档上下文规范」。审查时重点检查公共接口文档是否完整、是否同步更新。

### 维度9：Over-Engineering（过度工程）

借鉴 ponytail-debt 项目，专门针对过度工程的审查维度。

**审查标签**：
- **delete**：未使用的灵活性、投机性功能 → 直接删除（死代码见「冗余与质量审查」）
- **stdlib**：手写的标准库已有功能 → 使用标准库替代（见「重复造轮子」反模式）
- **native**：依赖或代码实现了平台已有的功能 → 使用原生特性
- **yagni**：单实现的抽象、无人设置的配置、单调用者的层 → 内联简化
- **shrink**：相同逻辑，更少行数 → 用更简洁的形式

**输出格式**：

```
<文件>:L<行号>: <tag> <问题描述>。<替换方案>。
```

**示例**：

```
src/validator.py:L12-38: stdlib: 27 行验证器类。"@" in email，1 行。
src/utils.ts:L4: native: moment.js 导入用于一个格式调用。Intl.DateTimeFormat，0 依赖。
src/repo.py:L88: yagni: AbstractRepository 只有一个实现。内联它直到第二个实现出现。
src/retry.ts:L52-71: delete: 幂等本地调用的重试包装。无需替换。
src/parser.py:L30-44: shrink: 手动循环构建字典。dict(zip(keys, values))，1 行。
```

**评分指标**：

```
net: -<N> lines possible.
```

如果无需删减：`Lean already. Ship.`

## 2. 代码质量量化标准

> 所有量化阈值的定义见 `coding_domain_rules.md` §九.3，此处仅列出审查判定规则。

### 2.1 复杂度阈值

> 所有阈值定义见「量化阈值」（§九.3），此处仅列出判定规则：
> - 圈复杂度超标：Must Fix
> - 认知复杂度超标：Should Fix
> - 嵌套层数超标：Must Fix
> - 函数行数超标：Must Fix

### 2.2 静态分析标准

- **重复率超标** \[error]: 超过阈值必须提取
- **依赖分析** \[warning]: 检查循环依赖、未使用依赖、过度依赖
- **安全漏洞** \[error]: 检测SQL注入、XSS、硬编码密钥等安全风险

## 3. Fallback 与兼容性审查

> 兼容性/Fallback 原则定义见 `coding_domain_rules.md` §三。本节仅列出审查判定规则。

### 适用范围 \[error]

- **降级滥用检测** \[error]: 是否对内部模块间调用设计了不必要的降级方案（内部调用不需要降级）
- **兼容范围检测** \[error]: 是否对内部接口/数据结构做了兼容处理（内部变更应直接统一升级，不保留旧路径）

### 降级合理性 \[warning]

- **外部依赖降级** \[warning]: 对外部依赖的降级路径是否明确、有意识
- **功能完整性** \[error]: 降级/兼容方案是否导致核心功能缺失或不可用
- **最终兜底** \[error]: 降级链的终点是否为明确报错（禁止假装一切正常）

### 兼容性安全性（仅限外部依赖） \[error]

- **破坏性变更** \[error]: 对外部暴露的接口是否存在删除字段、修改字段名、修改语义等破坏性变更
- **默认值** \[error]: 对外部暴露的新增字段/参数是否提供默认值
- **废弃流程** \[warning]: 移除外部接口是否遵循 Deprecate → Warn → Remove 流程

## 4. 审查报告规范

### 问题分级标准

- **Must Fix**: 必须修复才能通过审查（对应\[error]级别问题）
- **Should Fix**: 建议修复，不阻止合并但影响质量（对应\[warning]级别问题）
- **Nit**: 小问题，可选修复（代码风格、命名偏好等）

### 审查结论标准

- **Approve**: 代码质量合格，可以合并
- **Request Changes**: 存在Must Fix级别问题，需要修改后重新审查
- **Comment**: 仅提供建议，不阻止合并

### 报告格式要求

每个问题包含：

- 文件路径和行号
- 问题级别（Must Fix / Should Fix / Nit）
- 问题描述
- 修复建议

报告末尾包含：

- 问题统计摘要（按级别分类计数）
- 审查结论
