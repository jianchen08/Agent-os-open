# 代码审查流程规范

---

## 1. 审查执行流程

### 1.1 审查前准备

1. 获取本次审查的验收标准（AC 清单），如有
2. 识别代码类型（前端/后端/全栈），确定适用的审查维度
3. 识别项目的编码规范文件（lint 配置、style guide、convention 文档）

### 1.2 审查执行顺序

```
维度一：功能完整性 → 维度二：状态覆盖 → 维度三：交互完整性
→ 维度四：文案与提示 → 维度五：可访问性 → 维度六：安全与健壮性
→ 维度七：验收标准核对 → 维度八：模块边界完整性 → 总结
```

每个维度独立评分，最终汇总为审查结论。

## 2. 审查维度详解

### 维度零-A/B/C：法定审查步骤

> 定义见「法定审查步骤」（§0.1 需求追溯审查、§0.2 架构边界四问、§0.4 冗余与质量审查），此处不重复。

### 维度一：功能完整性

**检查目标**：代码是否实现了需求中描述的所有功能。

**检查流程**：

1. 对照需求文档或 AC 清单，逐条确认是否有代码实现
2. 对每个功能点，检查是否覆盖了正常路径和失败路径
3. 检查是否有未处理的边界场景

**判定标准**：

- 有 AC 清单时：每个 must 级别的 AC 必须有对应实现，缺失为 Must Fix
- 无 AC 清单时：从代码和需求描述中推断功能点，明显遗漏为 Should Fix

### 维度二：状态覆盖

**检查目标**：所有用户可感知的状态是否都有对应的 UI 表现。

**检查流程**：

1. 从代码中识别所有涉及异步操作、数据获取、表单提交的组件
2. 对每个组件，列出其所有可能的状态（idle/loading/success/error/empty 等）
3. 检查每个状态是否都有对应的 UI 处理

**判定标准**：

- 缺少 loading 状态反馈：Must Fix
- 缺少 error 状态处理：Must Fix
- 缺少 empty 状态展示：Should Fix
- 具体状态列表取决于组件功能，不是固定值

**检查方法**：

- 搜索异步操作（fetch、axios、useQuery 等），确认是否有 loading/error 处理
- 搜索列表渲染，确认空数据时的展示
- 搜索表单提交，确认提交中和失败后的处理

### 维度三：交互完整性

**检查目标**：用户交互是否有合理的反馈和防护。

**检查流程**：

1. 识别所有可交互元素（按钮、链接、输入框、弹窗）
2. 对每个交互，检查以下方面：

| 检查项 | 判定方法 |
| ------ | ------- |
| 操作反馈 | 异步操作是否有防重复提交机制 |
| 破坏性操作 | 删除/支付等是否有确认机制 |
| 操作可逆性 | 不可逆操作是否有明确提示 |
| 键盘可操作性 | 核心流程是否可通过键盘完成 |

**判定标准**：

- 缺少防重复提交：Must Fix（涉及数据一致性的操作）
- 缺少破坏性操作确认：Must Fix
- 键盘操作不便：Should Fix

### 维度四：文案与提示

**检查目标**：用户可见的文本是否清晰、具体、有帮助。

**检查流程**：

1. 检查所有错误提示：是否告知原因和解决方向
2. 检查所有按钮文案：是否使用动词描述行为
3. 检查所有加载文案：是否描述正在进行的操作
4. 检查所有空状态文案：是否引导用户下一步操作

**判定标准**：

- 模糊的错误提示（如"操作失败"）：Should Fix
- 缺少错误提示：Must Fix
- 具体的文案标准取决于项目已有的文案规范

### 维度五：可访问性

**检查目标**：应用对辅助技术是否友好。

**检查流程**：

1. 检查表单元素是否有标签关联（label / aria-label）
2. 检查错误提示是否通过 aria-describedby 等方式关联到输入框
3. 检查是否有仅依赖颜色传达的信息
4. 检查交互是否支持键盘操作

**判定标准**：

- 表单缺少标签关联：Should Fix
- 纯颜色传达关键信息：Should Fix
- 具体的可访问性标准参考项目已有的 a11y 配置

### 维度六：安全与健壮性

**检查目标**：代码是否处理了安全风险和异常情况。

**检查流程（后端）**：

1. 检查输入验证：所有外部输入是否经过校验
2. 检查权限控制：每个端点是否有权限检查
3. 检查敏感数据：响应和日志中是否过滤了敏感字段
4. 检查超时和重试：外部调用是否有合理的超时设置

**检查流程（前端）**：

1. 检查 XSS 风险：用户输入是否经过转义
2. 检查资源清理：组件卸载时是否清理定时器和事件监听
3. 检查内存泄漏：是否有闭包持有大对象的风险

**判定标准**：

- 缺少输入验证：Must Fix
- 缺少权限检查：Must Fix
- 资源未清理：Should Fix

### 维度七：验收标准核对

**检查目标**：如果任务有结构化的验收标准（AC 清单），逐条核对实现状态。

**检查流程**：

1. 读取任务附带的 AC 清单
2. 对每个 AC，在代码中找到对应的实现
3. 标记每个 AC 的状态：已实现 / 未实现 / 部分实现

**判定标准**：

- must 级别的 AC 未实现：Must Fix
- should 级别的 AC 未实现：Should Fix + 需说明原因

### 维度八：模块边界完整性

**检查目标**：代码是否遵循模块化设计，模块间通过公共接口通信，内部实现不暴露。

**检查流程**：

1. **公共接口识别**：找出每个模块对外暴露的公共 API（非 `_` 前缀的类和函数）
2. **内部实现隔离**：检查模块内部实现是否用 `_` 前缀标记，是否通过 `__all__` 或文档声明公共接口
3. **跨模块调用检查**：搜索 import 语句，确认跨模块引用是否只使用了对方的公共接口
4. **依赖方向检查**：画出模块间依赖关系图，检查是否存在循环依赖
5. **职责内聚检查**：检查每个模块的公共接口是否围绕同一职责领域，是否有不属于该模块的接口

**判定标准**：

| 检查项 | 级别 |
|--------|------|
| 直接导入其他模块的内部实现（`_` 前缀） | Must Fix |
| 存在循环依赖（A→B→A） | Must Fix |
| 模块职责混乱（一个模块承担多个不相关职责） | Should Fix |
| 公共接口缺少类型签名或文档字符串 | Should Fix |
| 内部实现未标记 `_` 前缀（但被外部使用） | Should Fix |

## 3. 快速审查清单

> 审查时对照以下七个维度快速扫描：设计层面（散点/边界/耦合）、功能正确性（边界/错误路径/死代码）、可读性（命名/函数长度/注释意图）、可测试性（依赖注入/全局状态/关键路径）、性能（N+1/大对象拷贝/内存泄漏）、安全（注入/加密/权限/硬编码/文件上传）、工程实践（目录结构/文档同步/配置提取）。
