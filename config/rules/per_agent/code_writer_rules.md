# Code Writer Agent 规则文件

## 第一部分：代码生成标准

> **编码核心原则**（命名、单一职责、DRY、KISS、错误处理等）见 `coding_domain_rules.md`。
> **SOLID 原则**见 `coding_domain_rules.md` §二。
> **兼容性/Fallback 原则**见 `coding_domain_rules.md` §三。
> **量化阈值**（唯一真相源）见 `coding_domain_rules.md` §九.3。

---

### 1. 组合优先继承 [warning]

优先使用组合模式而非继承。

**规则要求**：
- 优先通过组合实现代码复用
- 继承层级不超过两层
- 优先使用接口而非具体实现
- 优先使用委托而非继承

### 2. 类型注解 [error]

所有公共函数必须有类型注解。

**规则要求**：
- 参数必须有类型注解
- 返回值必须有类型注解
- 复杂数据结构使用泛型或自定义类型
- 避免使用 any 类型

### 3. 无副作用 [warning]

函数应避免隐式副作用。

**规则要求**：
- 函数应避免修改全局状态
- 避免修改传入参数
- 副作用函数必须通过命名明确表达
- 优先使用纯函数

### 4. 依赖注入 [warning]

通过参数注入依赖，不硬编码外部依赖。

**规则要求**：
- 外部依赖通过参数传入
- 使用依赖注入容器管理依赖关系
- 便于单元测试时替换实现
- 降低模块间的耦合度

## 第二部分：输出质量标准

### 1. 可读性 [error]

代码应自文档化，复杂逻辑必须有注释说明。

**规则要求**：
- 代码结构清晰，格式一致
- 复杂业务逻辑添加解释性注释
- 公共 API 添加文档注释
- 避免过度简化的缩写导致难以理解

### 2. 可维护性 [warning]

代码结构清晰，模块边界明确，低耦合。

**规则要求**：
- 模块边界清晰，职责单一
- 模块间通过定义良好的接口通信
- 相关代码按功能组织在一起
- 代码修改应局限在最小范围

### 3. 可测试性 [warning]

代码设计应便于测试，依赖可 Mock。

**规则要求**：
- 核心逻辑提取为独立函数
- 外部依赖通过接口抽象
- 避免硬编码难以测试的逻辑
- 提供测试所需的钩子方法

### 4. 外科手术式修改 [error]

只触碰必须触碰的代码，禁止"顺便"改进相邻代码。

**规则要求**：
- 每一行修改必须能追溯到用户请求
- 禁止顺手改注释、格式化、重命名未提及的变量
- 禁止重构未被要求重构的代码
- 相邻代码视为禁区，除非显式要求
- 发现需改进的代码时，记录在注释中而非直接修改

### 5. 透明完成 [error]

不确定时必须明确说出来，禁止静默完成。

**规则要求**：
- 任务未完全完成时，必须报告完成百分比和不确定的部分
- 禁止将模糊结果标记为"完成"或"成功"
- 验证未通过时，明确说明哪些验证通过、哪些未通过
- 宁可说"完成了 80%，剩下 20% 不确定"，也不要假装全部完成
