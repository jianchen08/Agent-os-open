# 前端设计一致性流程规范

## 1. 设计系统发现流程

编写前端代码前，**必须先执行以下发现步骤**，不能跳过：

### 步骤一：识别设计系统来源

1. 查找项目使用的组件库（读取 package.json 的 dependencies）
2. 查找项目的设计令牌文件（如 theme.css、tokens.json、tailwind.config 等）
3. 查找项目的全局样式文件（如 global.css、variables.css）
4. 查找项目已有的同类组件实现（搜索相似功能的组件）

### 步骤二：提取设计规范

从发现结果中提取：
- 间距体系：项目的间距基准值和梯度
- 颜色体系：项目的语义色变量和用法
- 字体体系：字号梯度、字重规范
- 组件库：项目已有哪些可复用组件
- 布局体系：断点、栅格、容器宽度

### 步骤三：记录并遵守

将提取到的规范作为本次编码的约束基准，所有样式决策必须以此为锚点。

---

## 3. 样式一致性流程

### 3.1 间距

1. 先从项目的设计令牌/全局样式中找到间距梯度
2. 所有间距值必须从项目定义的梯度中选取
3. 如项目未定义间距梯度，使用项目中最常见的间距倍数作为基准

### 3.2 颜色

1. 先找到项目的语义色变量（主色、成功色、警告色、错误色等）
2. 所有颜色必须通过项目的变量/token 引用
3. 如需新颜色，必须在设计令牌中新增变量后引用，禁止硬编码

### 3.3 字体

1. 先从项目中找到字号、字重的使用模式
2. 遵循项目已有的字体梯度，不引入新的字号或字重值

### 3.4 新增设计令牌的流程

当需要引入项目设计系统中不存在的新值时：

1. 在设计令牌文件中新增变量
2. 在组件中通过变量引用
3. 在代码注释中说明新增原因
4. code_reviewer 审查时重点检查新增令牌的合理性

---

## 4. 审查一致性检查流程

code_reviewer 在审查前端代码时，按以下流程检查设计一致性：

### 4.1 检查步骤

1. **复用检查**：是否有重新实现了项目已有组件功能？
2. **令牌检查**：是否有硬编码的间距、颜色、字号值？
3. **模式一致性**：新代码的风格是否与项目已有代码一致？
4. **组件 API 一致性**：新组件的 Props 设计是否与项目风格一致？

### 4.2 问题定级

| 问题类型 | 级别 |
|---------|------|
| 重新实现已有组件 | Must Fix |
| 硬编码样式值（项目有令牌系统时） | Must Fix |
| 样式模式与项目不一致 | Should Fix |
| 新组件 API 设计不规范 | Should Fix |
