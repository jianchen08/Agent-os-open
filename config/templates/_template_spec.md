# 产出物模板创建规范

本文档是 **agent_maker** 在为新 Agent 创建配套产出物模板时的参考规范。
定义了模板的标准结构、类型分类、命名规则、创建流程和质量检查标准。

---

## 一、模板类型分类

模板分为两种类型，创建前必须先确定类型，因为类型决定了模板的结构要求：

| 类型 | 名称 | 消费方式 | 评估指南 | 典型场景 |
|------|------|----------|----------|----------|
| **A 型** | 消费型 | 被下游 Agent 解析并作为输入使用 | 不需要 | 环境状态报告、修改任务链、回滚策略 |
| **B 型** | 报告型 | 作为最终产出物交付，供人或评估 Agent 审阅 | **必须有** | 调研报告、代码分析报告、方案文档、任务链 |

**判断方法**：

- 如果模板产出物会被另一个 Agent 读取并据此执行操作 → A 型
- 如果模板产出物是任务最终交付物，需要质量评估 → B 型
- 如果不确定，默认按 B 型处理（包含评估指南不会造成问题，但缺少会导致质量失控）

## 二、三段式结构规范

所有产出物模板必须遵循三段式结构：**头部说明** → **正文章节** → **评估指南**。

### 2.1 第一段：头部说明

头部说明使用 HTML 注释块（``）包裹，不会被渲染到最终产出物中。

**格式**：

```markdown
# {模板标题}

```

**各小节规则**：

| 小节 | 必填 | 内容要求 |
|------|------|----------|
| 模板是什么 | 是 | 一句话定义，说明模板的身份和用途 |
| 模板的作用 | 是 | 列举 3-5 条核心作用，每条用 "—" 连接简短说明 |
| 如何使用本模板 | 是 | 按步骤描述使用流程，步骤间有先后顺序 |
| 适用场景 | 是 | 列举 3-5 个适用场景，每个场景附简短说明 |
| 与其他模板的关系 | 否 | 仅当模板间存在上下游依赖时填写 |
| 占位符说明 | 否 | 仅当模板中使用了 {xxx} 占位符或 [必填]/[可选]/[按需] 标记时填写 |

### 2.2 第二段：正文章节

正文章节是模板的核心内容，定义产出物的具体结构。

#### 2.2.1 章节结构

每个章节遵循以下格式：

```markdown
---

## 章节名称 [必填/可选/按需/条件必填]

{章节内容模板，包含占位符}

```

#### 2.2.2 章节标注规则

| 标注 | 含义 | 使用场景 |
|------|------|----------|
| `[必填]` | 必须填写 | 核心信息章节，缺失则产出物不完整 |
| `[可选]` | 按需填写 | 补充信息章节，有则更好，无也不影响 |
| `[按需]` | 根据场景决定 | 特定条件下才需要填写（如"涉及数据库变更时必填"） |
| `[条件必填]` | 满足条件时必填 | 当某个前置条件满足时变为必填（如"存在缺失资源时必填"） |

#### 2.2.3 字段说明格式

每个章节内的字段说明使用 HTML 注释块包裹，包含：

1. **章节说明**：用 `2. **填写时机**（可选）：用 `> **填写时机**：` 说明何时填写
3. **填写要求**（可选）：用 `> **填写要求**：` 说明填写规范
4. **字段表格**：使用 Markdown 表格定义每个字段

字段表格的标准列：

| 列 | 必填 | 说明 |
|---|---|---|
| 字段/列名 | 是 | 字段名称 |
| 类型 | 是 | 数据类型：string / integer / number / boolean / list / table / enum |
| 必填 | 是 | 是 / 否 / 按需 |
| 说明 | 是 | 字段用途说明 |
| 示例/可选值 | 否 | 示例值或枚举值列表 |

#### 2.2.4 章节分隔

- 章节之间使用 `---` 分隔线
- 章节标题使用 `##` 二级标题
- 子章节使用 `###` 三级标题

### 2.3 第三段：评估指南

评估指南使用 HTML 注释块包裹，定义产出物的质量检查标准。

#### 2.3.1 何时需要评估指南

| 模板类型 | 是否需要 | 原因 |
|----------|----------|------|
| B 型（报告型） | **必须** | 报告型产出物需要质量评估 |
| A 型（消费型） | 不需要 | 消费型产出物由下游 Agent 解析，解析失败即为不合格 |
| 不确定 | 建议包含 | 包含评估指南不会造成问题 |

#### 2.3.2 评估指南格式

```markdown
## 评估指南 [可选]

```

#### 2.3.3 评估结论标准（统一）

所有模板使用相同的评估结论标准，不可自定义：

| 结论 | 判定条件 | 后续动作 |
|------|---------|---------|
| 通过 | 所有必填维度通过，无阻塞性问题 | 任务完成，产出物可用 |
| 有条件通过 | 必填维度通过，但存在改进建议 | 标注改进项，任务可继续 |
| 不通过 | 存在必填维度未通过，或缺失关键内容 | 退回修改，说明不通过原因 |

#### 2.3.4 检查维度定制原则

- 维度数量：3-5 个（不少于 3 个，不超过 5 个）
- 必填维度至少包含：
  - **完整性**：必填章节是否已填写
  - **格式规范**：表格结构、编号、Markdown 格式是否正确
- 可选维度根据模板特点从以下选取或自定义：
  - **准确性**：内容是否有依据、是否客观
  - **可行性**：方案/计划是否可执行
  - **一致性**：前后章节是否对应、数据是否匹配
  - **可操作性**：建议/方案是否具体可执行
  - **安全性**：是否有回滚/风险应对

## 三、命名和存放规范

### 3.1 文件命名

- **格式**：`{purpose}_template.md`
- **命名风格**：小写字母 + 下划线，语义明确
- **示例**：`research_report_template.md`、`execution_plan_template.md`、`rollback_strategy_template.md`
- **禁止**：驼峰命名、连字符、中文、空格

### 3.2 存放位置

- **目录**：`config/templates/`
- **绝对路径**：`config/templates/{purpose}_template.md`

### 3.3 在 Agent YAML 中引用

在 Agent 配置的 `static_vars.items` 中通过路径注入：

```yaml
static_vars:
  enabled: true
  items:
    - name: "{模板中文名称}"
      type: "path"
      path: "config/templates/{purpose}_template.md"
```

同时在 `deliverables` 中关联模板路径：

```yaml
deliverables:
  - name: "{产出物名称}"
    description: "{产出物描述}"
    template_source: "knowledge"
    template_name: "config/templates/{purpose}_template.md"
    output_path: "{{workspace}}/{output_filename}"
    type: "markdown"
    required: true
```

## 四、填写示例规范

填写示例是模板的重要组成部分，帮助使用者理解如何正确填写。

### 4.1 基本规则

1. **每个填写示例必须包含至少一个非代码场景**
   - 非代码场景：不涉及编程的场景，如市场调研、活动策划、流程设计
   - 目的：证明模板的通用性，避免使用者误以为模板只适用于技术场景

2. **多个场景示例用 `---` 分隔**
   - 第一个示例通常是代码/技术场景
   - 第二个示例是非代码/业务场景

3. **使用 `> ` 引用块包裹示例内容**
   - 填写示例放在 HTML 注释块内的 `> ` 引用块中
   - 示例内容应具体、真实、可直接参考

### 4.2 示例格式

```markdown

```

### 4.3 示例质量要求

- 示例内容必须具体，不能是抽象描述
- 示例中的数据必须合理、可信
- 示例应覆盖该章节的主要字段
- 每个表格类章节至少提供一个完整行的示例
- 每个多行文本章节至少提供一段真实示例文本

## 五、创建流程

创建产出物模板时，按以下步骤执行：

### 步骤 1：确定模板类型

根据产出物的消费方式判断模板类型：

| 判断依据 | 模板类型 |
|----------|----------|
| 产出物被下游 Agent 解析执行 | A 型（消费型） |
| 产出物是最终交付物，需评估质量 | B 型（报告型） |
| 不确定 | 默认 B 型 |

### 步骤 2：确定模板用途和适用场景

- 明确模板解决什么问题
- 列举 3-5 个适用场景
- 确认与其他模板的上下游关系

### 步骤 3：按三段式结构创建

1. **编写头部说明**：按 2.1 节格式填写
2. **设计正文章节**：
   - 确定章节列表和顺序
   - 每个章节标注 [必填]/[可选]/[按需]/[条件必填]
   - 设计每个章节的字段表格
   - 编写占位符模板
   - 编写填写示例（至少一个非代码场景）
3. **编写评估指南**（B 型必须，A 型不需要）：
   - 确定检查维度（3-5 个）
   - 定义每个维度的通过标准
   - 使用统一的评估结论标准

### 步骤 4：在 Agent YAML 中引用

1. 在 `static_vars.items` 中添加模板路径注入
2. 在 `deliverables` 中设置 `template_name` 指向模板路径

## 六、质量检查清单

模板创建完成后，逐项检查：

### 6.1 结构检查

- [ ] 模板包含完整的三段式结构（头部说明 + 正文章节 + 评估指南）
- [ ] 头部说明包含所有必填小节（模板是什么、模板的作用、如何使用本模板、适用场景）
- [ ] 正文章节使用 `---` 分隔线分隔
- [ ] 评估指南存在（B 型必须，A 型不需要）

### 6.2 章节检查

- [ ] 每个章节有明确的 [必填]/[可选]/[按需]/[条件必填] 标注
- [ ] 每个章节有章节说明（在 HTML 注释块中）
- [ ] 表格类章节有字段表格定义（在 HTML 注释块中）
- [ ] 多行文本类章节有内容建议说明

### 6.3 示例检查

- [ ] 每个填写示例至少包含一个非代码场景
- [ ] 多个场景示例使用 `---` 分隔
- [ ] 示例内容使用 `> ` 引用块包裹
- [ ] 示例内容具体、可信、可参考

### 6.4 评估指南检查（B 型模板）

- [ ] 检查维度数量在 3-5 个之间
- [ ] 包含"完整性"维度
- [ ] 包含"格式规范"维度
- [ ] 评估结论标准使用统一的三级标准（通过/有条件通过/不通过）
- [ ] 每个维度有明确的"通过标准"

### 6.5 引用检查

- [ ] 文件名符合 `{purpose}_template.md` 命名规范
- [ ] 文件存放在 `config/templates/` 目录下
- [ ] 对应的 Agent YAML 中已添加 `static_vars.items` 路径引用
- [ ] 对应的 Agent YAML 的 `deliverables` 中已设置 `template_name`

### 6.6 格式检查

- [ ] 头部说明和评估指南使用 HTML 注释块（``）包裹
- [ ] 正文字段说明使用 HTML 注释块包裹
- [ ] 占位符使用 `{xxx}` 格式
- [ ] 表格列对齐，Markdown 格式正确
- [ ] 无残留的示例数据或测试内容

## 七、常见问题

### Q1：模板中章节很多，如何决定哪些是必填？

**原则**：
- 核心信息章节 → 必填（如基本信息、摘要、结论）
- 补充说明章节 → 可选（如附录、术语表）
- 特定条件才需要的章节 → 按需（如数据模型仅在涉及数据库时需要）
- 满足某条件时才必填 → 条件必填（如缺失资源分析仅在存在资源缺口时需要）

### Q2：A 型和 B 型模板的核心区别是什么？

**核心区别**在于消费方式和评估要求：
- A 型：产出物被程序解析，格式正确性是关键，不需要评估指南
- B 型：产出物供人审阅或被评估 Agent 检查，内容质量是关键，必须有评估指南

### Q3：已有模板格式不完全一致怎么办？

本规范是对新模板的要求。遇到已有模板格式不一致的情况：
- 创建新模板时，严格遵循本规范
- 修改已有模板时，在不破坏兼容性的前提下逐步对齐本规范
- 不要为了统一格式而大规模重写已有模板
